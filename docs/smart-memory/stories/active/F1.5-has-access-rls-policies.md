# Story F1.5: Função has_access() + RLS em todas as tabelas

## Objetivo

Implementar a função `has_access(user_id, lesson_id)` (coração da autorização) e habilitar RLS com policies explícitas em **todas** as tabelas, conforme PRD Seções 5.3 e 5.4.

## Acceptance Criteria

- [ ] AC1: Função `public.has_access(p_user_id uuid, p_lesson_id uuid)` criada como `language sql security definer stable` com lógica exata do PRD Seção 5.3
- [ ] AC2: Função respeita `cardinality(included_module_ids) = 0` (vazio = todos) vs lista específica
- [ ] AC3: Função verifica `cm.status = 'ACTIVE'` E `(cm.expires_at IS NULL OR cm.expires_at > now())`
- [ ] AC4: RLS habilitado (`alter table ... enable row level security`) em todas as 21 tabelas do schema
- [ ] AC5: Policies de `profiles`: SELECT/UPDATE próprio; ADMIN tudo
- [ ] AC6: Policies de `courses`/`modules`/`lessons`: SELECT se published=true E `has_access()`; ADMIN escreve
- [ ] AC7: Policies de `materials`: SELECT se `has_access(auth.uid(), lesson_id)` da lesson relacionada
- [ ] AC8: Policies de `cohort_members`: usuário SELECT só onde `user_id = auth.uid()`; ADMIN SELECT all; INSERT/UPDATE só via service_role (webhook)
- [ ] AC9: Policies de `lesson_progress`: usuário CRUD próprias linhas; ADMIN SELECT all
- [ ] AC10: Policies de `comments`: SELECT se `has_access()`; INSERT autenticado; UPDATE/DELETE pelo autor; ADMIN qualquer
- [ ] AC11: Policies de `forum_*`: leitura por qualquer usuário com qualquer cohort_member ACTIVE
- [ ] AC12: Policies de `payments`: usuário SELECT próprios; ADMIN SELECT all; service_role escreve
- [ ] AC13: `webhook_events`: apenas service_role lê/escreve (sem policy para anon/authenticated)
- [ ] AC14: Testes manuais com 3 usuários (admin, student matriculado em cohort A, student não matriculado) cobrem casos: ler lesson liberada, ler lesson bloqueada, ler material, ler payment próprio, ler payment alheio (deve falhar)
- [ ] AC15: Documentação em `docs/smart-memory/agents/data-engineer/rls-policies.md` listando policy a policy

## Escopo

**IN:**
- Função `has_access()` em SQL
- RLS habilitado em todas as 21 tabelas
- Policies completas conforme PRD 5.4
- Documentação das policies
- Suite de testes manuais (script SQL ou Studio queries)

**OUT:**
- Testes automatizados de RLS (Fase 8)
- Helpers TS (`requireAccess`, etc) — vão na Fase 2

## Contexto Técnico

- `has_access()` é usada em policies E na aplicação (Server Components) — **é o coração da autorização**
- `security definer` permite a função consultar tabelas mesmo quando o usuário não tem SELECT direto
- `stable` permite cache em planner
- ATENÇÃO: policies devem tratar caso `auth.uid() IS NULL` (anon) — geralmente recusar
- ATENÇÃO: ADMIN check em todas policies — usar `(SELECT role FROM profiles WHERE id = auth.uid()) = 'ADMIN'`
- Performance: índice composto em `cohort_members(user_id, status)` é crítico para `has_access()` ser rápido
- Risco crítico (ADR-001): RLS mal configurado vaza dados; auditoria pré-Fase 2 obrigatória
