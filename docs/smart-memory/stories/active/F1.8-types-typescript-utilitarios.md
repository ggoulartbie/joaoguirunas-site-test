# Story F1.8: Geração de tipos TS + utilitários de banco

## Objetivo

Estabelecer fluxo de geração de tipos TypeScript do schema e criar pasta `/lib/supabase/` com clients (server, browser, admin) usando os tipos gerados.

## Acceptance Criteria

- [ ] AC1: Script `npm run db:types` em `package.json` que executa `supabase gen types typescript --project-id mksmmpfyqowuzjcchhkl > types/database.ts`
- [ ] AC2: `types/database.ts` versionado e atualizado com schema atual da Fase 1
- [ ] AC3: `lib/supabase/server.ts` exporta `createServerClient()` para Server Components/Actions usando `@supabase/ssr` e cookies
- [ ] AC4: `lib/supabase/client.ts` exporta `createBrowserClient()` para Client Components
- [ ] AC5: `lib/supabase/admin.ts` exporta `createAdminClient()` usando `SUPABASE_SERVICE_ROLE_KEY`; arquivo marcado com `import 'server-only'` no topo
- [ ] AC6: Todos os clients tipados com `Database` de `types/database.ts` (genérico passa por toda função tipada)
- [ ] AC7: Type aliases utilitários em `types/db.ts`: `Profile`, `Course`, `Module`, `Lesson`, `Cohort`, `CohortMember`, `Payment`, `Coupon`, etc — extraídos de `Database['public']['Tables'][...]['Row']`
- [ ] AC8: README com seção "Atualizar tipos" explicando quando rodar `db:types` (após cada migration)

## Escopo

**IN:**
- Script de geração de tipos
- 3 clients Supabase
- Type aliases utilitários
- `server-only` no admin client (proteção runtime)

**OUT:**
- Helpers de auth (`requireUser`, `requireAdmin`) — Fase 2
- Helpers de domínio (createCheckoutSession, etc) — fases respectivas

## Contexto Técnico

- `@supabase/ssr` (não `@supabase/auth-helpers-nextjs` que está deprecated)
- `SUPABASE_SERVICE_ROLE_KEY` ignora RLS — `import 'server-only'` impede import acidental em Client Component (build quebra)
- Tipos gerados não incluem RPC functions automaticamente; `has_access` usado via SQL embutido ou helper
