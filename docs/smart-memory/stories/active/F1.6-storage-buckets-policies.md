# Story F1.6: Buckets de Storage + policies

## Objetivo

Criar os 4 buckets de Supabase Storage (`avatars`, `materials`, `certificates`, `covers`) com policies que respeitam a regra de acesso da plataforma.

## Acceptance Criteria

- [ ] AC1: Bucket `avatars` criado como **público** com policy de upload pelo próprio usuário (path `{user_id}/avatar.{ext}`)
- [ ] AC2: Bucket `covers` criado como **público** com upload restrito a ADMIN (capas de courses e cohorts)
- [ ] AC3: Bucket `materials` criado como **privado**; SELECT via signed URL com expiração de 5 minutos
- [ ] AC4: Bucket `certificates` criado como **privado**; SELECT via signed URL com expiração de 1 hora
- [ ] AC5: Policies de Storage em `materials` validam que usuário tem `has_access()` na lesson cujo material está sendo baixado (path convencionado: `materials/{lesson_id}/{material_id}.{ext}`)
- [ ] AC6: Policies de Storage em `certificates` validam que `auth.uid() = user_id` extraído do path (`certificates/{user_id}/{certificate_id}.pdf`)
- [ ] AC7: Migration SQL versionada com `insert into storage.buckets ...` e policies; aplicada via `supabase db push`
- [ ] AC8: Documentação em `docs/smart-memory/agents/data-engineer/storage.md` com paths convencionais e exemplos de signed URL

## Escopo

**IN:**
- Criação dos 4 buckets via SQL (não via painel)
- Policies de upload/download por bucket
- Convenção de paths documentada

**OUT:**
- Helpers TS de upload/signed URL (Fases 2-4)
- Watermark dinâmico em vídeos (Fase 3)

## Contexto Técnico

- PRD Seção 4.5: 4 buckets
- Path convention é load-bearing porque policies extraem IDs do path via `storage.foldername(name)` ou regex
- `materials`: signed URL 5min — material grande, uso pontual
- `certificates`: signed URL 1h — pode compartilhar; verificação pública usa rota separada `/certificado/v/[code]`
- Avatar: público porque é exibido em comments, fórum, perfil de outros alunos
