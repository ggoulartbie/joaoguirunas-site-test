# Story F13.4: Aviso no campo link estático + guia de turma de teste R$1

## Objetivo
Alertar o admin, dentro do `CohortForm`, sobre o trade-off de usar `infinitepay_checkout_url` (link estático) — que NÃO rastreia pagamentos via webhook — e produzir um guia operacional passo-a-passo para criar uma turma de teste de R$1 usando o fluxo dinâmico, incluindo a configuração do webhook no painel InfinitePay.

## Acceptance Criteria

- [ ] **AC1** — Em `src/app/(academy)/academy/(admin)/admin/turmas/CohortForm.tsx` (bloco entre lines 1000–1015), logo abaixo do `<input>` do campo `infinitepayCheckoutUrl` (ou logo abaixo do `<p>` explicativo existente, antes do `</div>` que fecha o campo), adicionar um `<Alert>` (ou componente equivalente do design system local — verificar `src/components/ui/`; se não existir, usar `<div>` com classes Tailwind padronizadas no admin, ex.: `border border-[var(--ember)] bg-[var(--ember)]/10 p-3 text-xs text-[var(--bone)]`) com o texto:
  > **Atenção:** Link estático não rastreia pagamentos automaticamente. O webhook não conseguirá associar o pagamento ao aluno. Use apenas se o checkout for externo à plataforma.
- [ ] **AC2** — O elemento de aviso possui `data-testid="infinitepay-static-url-warning"` para rastreabilidade em testes E2E e na auditoria visual do QA.
- [ ] **AC3** — Criar `docs/smart-memory/ops/infinitepay-setup-guide.md` com seções obrigatórias:
  1. **Pré-requisitos:** conta InfinitePay ativa. Env var `INFINITEPAY_HANDLE` configurada **via CLI** (nunca via dashboard Vercel):
     - **Produção (Vercel):** `vercel env add INFINITEPAY_HANDLE production` (e responder com o handle quando solicitado). Repetir com `preview` se quiser cobrir previews.
     - **Desenvolvimento local:** adicionar `INFINITEPAY_HANDLE=growthsales` em `.env.local` (nunca comitar — está no `.gitignore`).
     - **Nota:** InfinitePay NÃO usa `Authorization: Bearer` — autenticação é via campo `handle` no body. Ver runbook `docs/smart-memory/runbooks/checkout-infinitepay.md`. Não há `INFINITEPAY_BEARER_TOKEN` para configurar.
  2. **Passo 1 — Criar curso/turma de teste:** no admin `/academy/admin/turmas/novo`, criar cohort com `slug: turma-teste-r1`, `price_cents: 100` (R$1,00), `payment_provider: INFINITEPAY`, `infinitepay_handle: <handle-da-conta>`, **deixar `infinitepay_checkout_url` VAZIO** (fluxo dinâmico).
  3. **Passo 2 — Configurar webhook no painel InfinitePay:** URL exata `https://joaoguirunas.com/api/webhooks/infinitepay`. Sem assinatura HMAC (InfinitePay não suporta) — segurança via `payment_check` server-to-server já implementado. Este é o **único** passo do guia que envolve clique em painel de terceiro (InfinitePay), porque não há CLI/API InfinitePay para isso.
  4. **Passo 3 — Fluxo de teste end-to-end:** logar como aluno → ir para `/turmas/turma-teste-r1` → clicar em comprar → ser redirecionado ao checkout InfinitePay → pagar com cartão de teste (documentar onde encontrar) → confirmar redirect de sucesso → verificar em `/academy/admin/pagamentos` que o `payment` aparece como `APPROVED` → verificar em `/academy/admin/usuarios` que o aluno aparece como `cohort_members.status = ACTIVE` na turma → verificar que o aluno tem acesso ao curso em `/academy/meus-cursos`.
  5. **Passo 4 — Checklist de validação (formato `- [ ]`)** com cada ponto acima e mais: webhook recebido (verificar logs via `vercel logs --follow` ou `vercel logs <deployment-url>`), `order_nsu` bate com `payments.id`, email transacional de confirmação enviado, registros no Supabase verificáveis via `supabase` CLI ou Studio (`payments` row com `status=APPROVED`, `cohort_members` row com `status=ACTIVE`).
  6. **Troubleshooting comum:** pagamento ficou `PENDING` (verificar logs do `/payment_check` via `vercel logs`); warning Sentry de `order_nsu sem payment` (provavelmente usou link estático — ver F13.4 AC1/AC2); env var ausente em produção (verificar com `vercel env ls production` — nunca via dashboard).
- [ ] **AC4** — O guia inclui menção explícita à URL `https://joaoguirunas.com/api/webhooks/infinitepay` na seção do Passo 2, em bloco `code` para copy/paste sem ambiguidade. Adicionalmente, todos os comandos de env/logs aparecem em blocos `code` shell (ex.: `vercel env add INFINITEPAY_HANDLE production`, `vercel env ls production`, `vercel logs --follow`).

## Contexto Técnico

**Arquivos afetados:**
- `src/app/(academy)/academy/(admin)/admin/turmas/CohortForm.tsx` (adicionar `<Alert>` em ~line 1014, antes do `</div>` que fecha o bloco do campo)
- `docs/smart-memory/ops/infinitepay-setup-guide.md` (novo)
- `docs/smart-memory/INDEX.md` (verificar existência; se existir, adicionar wikilink)

**Estado atual:**
- Campo `infinitepay_checkout_url` em `CohortForm.tsx` (lines 1001–1014) já existe com label "URL de Checkout (link estático InfinitePay)" e descrição funcional, mas SEM aviso sobre o trade-off de não rastrear pagamentos.
- Webhook URL canônica: `https://joaoguirunas.com/api/webhooks/infinitepay` (rota pública, sem HMAC).
- Fluxo dinâmico (campo vazio) já funcional: cria PENDING, envia `order_nsu`, webhook resolve via `/payment_check`.
- Fluxo estático (campo preenchido) é o gap: redireciona direto ao link externo sem criar PENDING → webhook chega e não acha payment.

**Dependências:**
- Nenhuma. A F13.4 é autocontida; o adapter `src/lib/payment/infinitepay.ts` já está correto (autentica via `handle` no body — não há gap de Bearer Token a corrigir).

**Constraints:**
- O `<Alert>` deve ser sempre visível quando `paymentProvider === 'INFINITEPAY'` (ele já está dentro do bloco condicional na line 1000) — não é necessário condicionar à existência de valor no campo. O admin precisa ver o aviso *antes* de preencher.
- Não usar emoji no `<Alert>` (regra global do projeto: "avoid emojis unless explicitly requested").
- O guia em markdown segue padrão Obsidian com frontmatter + wikilinks (ver `docs/smart-memory/ops/launch-readiness.md` como referência de estilo).
- **Infra via CLI, nunca via dashboards:** todo passo de configuração de env var deve usar `vercel env add/ls/rm` (produção) ou `.env.local` (dev). Não documentar cliques no dashboard Vercel. Configurações de Supabase devem usar `supabase` CLI quando aplicável. A única exceção é o painel da própria InfinitePay para configurar o webhook URL (não há CLI deles).

**Decisões:**
- Aviso textual, não bloqueio: o link estático pode ser legitimamente usado quando o produto não é entregue pela plataforma (ex.: mentoria 1:1 que vende fora do gate de cohort). Bloquear seria over-engineering.
- Preço da turma de teste: definir no guia como **R$1,00 (`price_cents: 100`)** — InfinitePay tem mínimo prático de R$1,00 para captura em cartão; centavos podem ser rejeitados.
- `data-testid` em vez de `id` ou classe: padrão do projeto para QA, não polui CSS.
- Convenção CLI-first para env vars: João prefere `vercel env add` e `.env.local` em vez de cliques no dashboard Vercel — evita drift entre ambientes e permite versionar conhecimento operacional em scripts/runbooks.
