# Story FAA-1.4: Fix Vimeo iframe dimensions v2 — regressão em vídeos não-16:9

## Objetivo

Forçar o iframe injetado pelo Vimeo Player SDK a preencher 100% do container 16:9
(via CSS no wrapper, não via `responsive: true` do SDK), eliminando a regressão
do bug fixado em FAA-1.1 que voltou a aparecer em vídeos cujo aspect ratio
nativo difere de 16:9.

## Por que recorreu

A FAA-1.1 confiou no `responsive: true` do `@vimeo/player` SDK assumindo que ele
faria o iframe preencher o container 16:9. Na prática, o modo `responsive: true`
respeita o **aspect ratio nativo do vídeo** (lido via API após o `ready()`) e
ajusta a altura do iframe proporcionalmente à largura do container. Resultado:

- Vídeo 16:9 dentro de container 16:9 → preenche corretamente (caso testado em FAA-1.1).
- Vídeo 9:16 (vertical) dentro de container 16:9 → o SDK estica a altura do
  iframe para manter 9:16, gerando overflow vertical e cortando a barra de controles.
- Vídeo 4:3 dentro de container 16:9 → letterbox / overflow lateral similar.

A modificação local do João em `feat-aulas-v2` (não commitada) já testou
empiricamente a hipótese correta: desligar o `responsive` do SDK (`responsive: false`)
e forçar via CSS Tailwind o iframe filho a `position: absolute; inset: 0;
width: 100%; height: 100%`. Isso garante que **o container 16:9 é fonte da
verdade** — o vídeo é renderizado dentro dele com `object-fit`-equivalente do
iframe (Vimeo internamente preserva o aspect ratio nativo no conteúdo, mas o
iframe em si fica do tamanho do slot).

Trade-off aceito: vídeos não-16:9 terão letterbox/pillarbox **dentro** do
iframe (barras pretas dos lados ou cima/baixo), mas a barra de controles
fica sempre visível e nenhum overflow vaza do container. Esse é o
comportamento desejado para uma área de aulas onde consistência visual e
acesso aos controles importam mais do que maximizar área do vídeo.

**Abordagem aprovada (decisão arquitetural):**

Manter `responsive: false` no SDK Vimeo e controlar o sizing 100% via CSS no
wrapper. Razão técnica: o `responsive: true` do SDK delega o cálculo de
altura para o JS interno do Vimeo, que usa o aspect ratio nativo do vídeo —
isso é exatamente o comportamento que queremos suprimir. Com `responsive:
false` o SDK injeta o iframe sem `width`/`height` inline próprios, e o CSS
do pai dita as dimensões. A regra Tailwind `[&_iframe]:...` é um seletor
descendente arbitrário do Tailwind v4 que casa o `iframe` direto injetado
pelo SDK dentro do `vimeoContainerRef`, sem precisar passar a ref dele para
o iframe (que não temos).

**Trade-off documentado:** vídeos não-16:9 mostrarão letterbox/pillarbox
dentro do player Vimeo (barras pretas internas mantidas pelo próprio Vimeo).
Isso é aceitável e preferível a vazar o iframe para fora do container, porque
(a) os controles ficam acessíveis, (b) a UI da página de aula mantém
consistência visual, (c) o aluno pode dar fullscreen se quiser ver o vídeo
no aspect ratio nativo.

**Alternativa considerada e descartada:** detectar o aspect ratio do vídeo
via `player.getVideoWidth()/getVideoHeight()` no callback `ready()` e ajustar
o `aspectRatio` do container dinamicamente. Descartada porque:
1. Causa layout shift visível (CLS ruim) entre o estado inicial 16:9 e o
   ajustado.
2. Quebra a grid da página de aula (cards laterais reposicionam).
3. Não tem como acontecer no SSR, então o primeiro paint é sempre 16:9 e
   depois "salta".

**Por que não confiar só no SDK:** o `responsive: true` do `@vimeo/player`
não é configurável para "preencher o container ignorando aspect ratio nativo"
— é um comportamento all-or-nothing. A documentação do SDK confirma:
`responsive` faz o iframe redimensionar para manter o aspect ratio nativo.
Forçar via CSS é o pattern recomendado quando você quer aspect ratio fixo
de container.

**Critério de "pronto":** AC2 cumprido com prints/observações dos 3 vídeos no
QA Results — não basta "funciona no meu vídeo de teste". O dev deve obter
3 `video_id`s reais (do próprio João ou subir 2 clipes de teste vertical/4:3
no painel Vimeo) antes de fechar a story. Se AC2 não puder ser cumprido por
indisponibilidade de vídeos não-16:9, escalar ao lead via SendMessage com
proposta de critério alternativo.

**Risco:** muito baixo. Mudança CSS + um toggle de boolean no SDK. Sem
impacto em lógica de progresso, refs ou efeitos.

### Validações adversariais realizadas (além dos ACs)

1. **Risco: `responsive: false` quebra eventos do SDK?** — Auditado em `node_modules/.pnpm/@vimeo+player@2.30.4/node_modules/@vimeo/player/dist/player.es.js`. `responsive` é listado nos `oEmbedParameters` (linha 818): é apenas um parâmetro de URL do iframe Vimeo, **completamente desacoplado** da camada de eventos `postMessage` que alimenta `play/pause/ended/timeupdate`. **Risco zero** de quebrar `saveProgress`/`markLessonComplete`.

2. **Risco: Tailwind v4 não suporta `[&_iframe]:*`?** — Confirmado suporte. `[&_selector]:utility` é o padrão de arbitrary variant nativo do Tailwind v3.2+ e mantido no v4. Selector descendente — funciona mesmo se Vimeo SDK injetar wrapper extra entre `vimeoContainerRef` e o iframe.

3. **Risco: regressão no consumer?** — `src/app/(academy)/academy/(student)/curso/[slug]/aula/[lesson-slug]/page.tsx:349` já envolve o `<VideoPlayer>` num `<div className="aspect-video w-full shrink-0">` SEM passar `className` para o componente (prop default `undefined`). Isso resulta em aninhamento de 3 divs (aspect-video externo → div vazio → div com aspect-ratio interno). **Não é bug**, mas é débito visual menor — recomendação opcional: passar `className="h-full w-full"` no consumer ou remover o `<div className={className}>` interno. **Não-bloqueante.**

4. **Risco: outros consumidores afetados?** — `grep "VideoPlayer"` em `src/`: único consumer ativo é a página de aula (`[lesson-slug]/page.tsx`). `LessonEditorClient.tsx` importa apenas `extractYouTubeId` (helper, não tocado). **Sem outros impactos.**
