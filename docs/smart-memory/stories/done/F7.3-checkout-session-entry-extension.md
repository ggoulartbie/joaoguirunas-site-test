# Story F7.3: createCheckoutSession + decisão automática ENTRY vs EXTENSION

## Objetivo

Server Action que cria sessão Stripe Checkout para uma cohort, decidindo automaticamente o preço (entry para novo aluno, extension para aluno com histórico em `cohort_members`).

## Acceptance Criteria

- [ ] AC1: Server Action `createCheckoutSession(cohortSlug, couponCode?)` em `app/actions/checkout.ts`:
  - Valida usuário autenticado (chama `requireUser`)
  - Carrega cohort por slug; valida `is_purchasable=true`; valida vagas se `total_seats` definido
  - Verifica se user **já tem `cohort_members`** nessa cohort (qualquer status)
    - Sim → `purchase_kind=EXTENSION`, usa `stripe_price_extension_id`
    - Não → `purchase_kind=ENTRY`, usa `stripe_price_entry_id`
  - Aplica cupom se `couponCode`: valida `applies_to`, `is_active`, validade temporal, `current_uses < max_uses`
  - Cria/recupera `stripe_customer_id` e salva em `profiles`
  - Cria Stripe Checkout Session com `metadata`: `user_id`, `cohort_id`, `purchase_kind`, `coupon_id?`
  - Retorna `url` do checkout
- [ ] AC2: `app/(public)/checkout/[cohortSlug]/page.tsx` (Server Component) chama o action e redireciona pra URL
- [ ] AC3: `app/(public)/checkout/sucesso/page.tsx` mostra "Compra concluída! Acessando seu painel..." e redireciona para /dashboard após 3s
- [ ] AC4: `app/(public)/checkout/cancelado/page.tsx` mostra "Compra não concluída" com CTA para voltar à LP da cohort
- [ ] AC5: Tratamento de erros: cohort não vendável → 404; sem vagas → mensagem amigável; cupom inválido → mensagem
- [ ] AC6: Botão "Quero entrar" da LP `/turmas/[slug]` chama esse action

## Escopo

**IN:**
- Server Action de checkout
- 3 páginas de checkout
- Lógica de decisão entry/extension
- Validação de cupom

**OUT:**
- Webhook handler (F7.4)
- Aplicação de cross extensions (F7.4)
- Auto renewal (F7.5)

## Contexto Técnico

- ADR-001 D11: elegibilidade para extension é vitalícia — buscar em `cohort_members` mesmo com `status=REMOVED` ou `EXPIRED`
- ADR-001 D12: recompra antes de expirar soma ao prazo — mas isso é responsabilidade do webhook (F7.4); checkout só decide qual price usar
- Metadata é load-bearing: webhook usa pra criar `cohort_members` corretamente
- Cupom: `applies_to` deve bater com `purchase_kind` decidido (ENTRY/EXTENSION/BOTH)
