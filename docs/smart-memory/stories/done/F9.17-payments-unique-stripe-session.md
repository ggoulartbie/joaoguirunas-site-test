# Story F9.17: UNIQUE constraint em payments(stripe_checkout_session_id) + ON CONFLICT no webhook

## Objetivo
Garantir idempotência do webhook Stripe ao nível do banco: criar UNIQUE constraint em `payments(stripe_checkout_session_id)` e converter os INSERTs do webhook para `ON CONFLICT DO NOTHING`, prevenindo duplicação de registos de pagamento quando o Stripe reenvia events com IDs diferentes para a mesma session.

## Acceptance Criteria
- [ ] AC1: Migration `supabase/migrations/{timestamp}_payments_unique_session.sql` cria índice único: `CREATE UNIQUE INDEX IF NOT EXISTS payments_stripe_checkout_session_id_key ON payments(stripe_checkout_session_id) WHERE stripe_checkout_session_id IS NOT NULL;` (partial index — permite múltiplos rows com session_id NULL para pagamentos de outras origens, ex: renovação automática sem checkout).
- [ ] AC2: Migration roda sem falhar — antes de criar o índice, deduplica linhas existentes via CTE: manter o registo mais antigo (menor `created_at`) por `stripe_checkout_session_id`, deletar os demais. Logar quantas linhas foram removidas via `RAISE NOTICE`.
- [ ] AC3: As 4 chamadas `supabaseAdmin.from('payments').insert({...})` em `src/app/api/webhooks/stripe/route.ts` (linhas ~264, ~385, ~472, ~543) passam a usar `.upsert({...}, { onConflict: 'stripe_checkout_session_id', ignoreDuplicates: true })` OU validam previamente via `SELECT id FROM payments WHERE stripe_checkout_session_id = ?` antes do insert. Decidir padrão único e aplicar em todas as 4.
- [ ] AC4: Smoke test manual em ambiente staging: enviar mesmo event `checkout.session.completed` 2x (via Stripe CLI `stripe trigger` + replay) e verificar que `SELECT count(*) FROM payments WHERE stripe_checkout_session_id = '<id>'` retorna 1, não 2. Documentar comando exato no PR description.
- [ ] AC5: Tipos TS regenerados (`pnpm db:types` ou equivalente) após a migration — confirmar que nenhum lugar quebra. `tsc --noEmit` passa.
- [ ] AC6: Decisão documentada no header da migration: "Idempotência ao nível DB. Stripe pode reenviar webhooks; constraint aplicação não é suficiente porque cross-tab e race conditions tornam o check `existingMember` insuficiente para `payments`."

## Escopo

**IN:**
- 1 migration nova em `supabase/migrations/` com partial unique index + dedupe CTE
- Modificação dos 4 INSERTs em `src/app/api/webhooks/stripe/route.ts`

**OUT:**
- Refactor mais amplo do webhook (escopo F7.4 / F9.10)
- Adicionar UNIQUE em outras tabelas (cohort_members já protegido por check `existingMember`; lesson_progress tem PK composta — fora de escopo)
- Mudança de schema de payments (apenas adicionar constraint)
- Reprocessamento de pagamentos históricos duplicados em produção — apenas documentar quantidade

## Contexto Técnico
- Webhook em `src/app/api/webhooks/stripe/route.ts` chama `payments.insert()` em 4 paths: ENTRY checkout, EXTENSION checkout, recurring invoice, subscription update.
- `stripe_checkout_session_id` pode ser NULL para pagamentos sem checkout session (ex: cron de renovação automática) — daí o partial index `WHERE stripe_checkout_session_id IS NOT NULL`.
- Memberships protegidos por `existingMember` check; gap atual é apenas em `payments` rows.
- Migration timestamp: usar formato projeto (`YYYYMMDDHHMMSS_descricao.sql` — referência: `20260506022037_has_access_rls_policies.sql`).
- Risco: dedupe pode remover linhas legítimas se duas sessions Stripe distintas compartilharem `stripe_checkout_session_id` (impossível por design Stripe, mas confirmar antes do dedupe rodar em prod).
- Squads sugeridos: sites-data faz a migration; sites-dev-beta atualiza o webhook (já familiar com o ficheiro pela auditoria F9.5).
