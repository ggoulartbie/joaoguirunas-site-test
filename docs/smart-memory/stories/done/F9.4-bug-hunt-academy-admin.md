# Story F9.4: Bug hunt sweep nas páginas /academy/admin/*

## Resultado da Auditoria

### Bugs Encontrados

#### B-01 — P0 CRÍTICO — Botão de reembolso silencioso
**Página:** `/academy/admin/pagamentos`
**Arquivo:** `PaymentsClient.tsx` + `pagamentos/actions.ts`
**Descrição:** `handleRefundConfirm` só fechava o modal — não chamava nenhuma Server Action. O reembolso nunca era processado no Stripe e o status do pagamento nunca era atualizado.
**Status:** CORRIGIDO nesta story.
**Fix:**
- Adicionada `refundPayment(paymentId)` em `pagamentos/actions.ts` — chama `getPaymentAdapter().refundPayment()`, atualiza status para `REFUNDED`, marca `cohort_member` como `REMOVED`.
- `PaymentsClient.tsx` importa e chama a action, exibe loading state e erro no modal.

#### B-02 — P2 — CourseListClient sem error handling em toggle/delete
**Página:** `/academy/admin/cursos`
**Arquivo:** `CourseListClient.tsx`
**Descrição:** `handleToggle` e `handleDelete` fazem otimistic update mas ignoram erros da Server Action. Se a action falhar, a UI fica inconsistente sem feedback ao admin.
**Status:** Documentado. P2 — não bloqueia funcionalidade crítica.

#### B-03 — P2 — ForumModerationClient sem error handling
**Página:** `/academy/admin/forum`
**Arquivo:** `ForumModerationClient.tsx`
**Descrição:** `handleThreadAction` e `handleReplyAction` chamam as actions dentro de `startTransition` sem try/catch. Erro silencioso se a moderação falhar.
**Status:** Documentado. P2.

#### B-04 — P3 — Campos de URL de live sessions sem validação client-side
**Página:** `/academy/admin/turmas/[id]`
**Arquivo:** `CohortForm.tsx`
**Descrição:** Campos `meetingUrl` e `recordingUrl` são `TextInput` sem validação de formato de URL no cliente. A validação existe no server (Zod), mas usuário só vê erro após submit.
**Status:** Documentado. P3 — server valida corretamente. UX menor.
