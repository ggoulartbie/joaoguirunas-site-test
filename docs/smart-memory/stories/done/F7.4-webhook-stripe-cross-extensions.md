# Story F7.4: Webhook Stripe + idempotência + cross extensions

## Objetivo

Implementar `/api/webhooks/stripe` com idempotência via `webhook_events`, tratando os 4 eventos críticos (`checkout.session.completed`, `invoice.paid`, `invoice.payment_failed`, `charge.refunded`) e aplicando cross extensions.

## Acceptance Criteria

- [ ] AC1: Route Handler `app/api/webhooks/stripe/route.ts` (POST) verifica assinatura via `STRIPE_WEBHOOK_SECRET`
- [ ] AC2: Webhook usa `createAdminClient()` (service_role) — bypassa RLS
- [ ] AC3: **Idempotência**: antes de processar, INSERT em `webhook_events` com `stripe_event_id` UNIQUE; se conflito, retorna 200 imediatamente (já processado)
- [ ] AC4: Em caso de erro, atualiza `webhook_events.success=false` + `error_message`; retorna 500 (Stripe re-tenta)
- [ ] AC5: Handler `checkout.session.completed`:
  - Lê `metadata` (user_id, cohort_id, purchase_kind, coupon_id?)
  - Cria `payments` (status=APPROVED, amount_cents, stripe_*_id)
  - Se `purchase_kind=ENTRY`:
    - Cria `cohort_members` com `expires_at = now() + access_duration_days` (ou NULL se vitalício); status=ACTIVE
    - Incrementa `cohorts.filled_seats`
  - Se `purchase_kind=EXTENSION`:
    - Atualiza `cohort_members` existente: `expires_at = max(expires_at, now()) + extension_duration_days`; status=ACTIVE
  - **Aplica cross extensions**: para cada `cohort_cross_extensions` com `source_cohort_id = cohort_id` e `is_active=true`:
    - Se user tem `cohort_members` no `target_cohort_id`: estende `expires_at += days_granted`
    - Se não tem: cria `cohort_members` ACTIVE com `expires_at = now() + days_granted`
  - Incrementa `coupons.current_uses` se aplicado
  - Vincula `payments.membership_id`
  - Dispara email "Acesso liberado" (template em F7.6)
- [ ] AC6: Handler `invoice.paid`:
  - Caso renovação automática (`stripe_subscription_id` presente):
    - Cria `payments` com `purchase_kind=AUTO_RENEWAL`
    - Estende `cohort_members.expires_at += extension_duration_days`
- [ ] AC7: Handler `invoice.payment_failed`:
  - Marca `cohort_members.status='PAST_DUE'`
  - Após 3 falhas (lógica em metadata ou contador): desativa renovação automática + status=EXPIRED
- [ ] AC8: Handler `charge.refunded`:
  - Atualiza `payments.status='REFUNDED'`
  - Decisão: manter membership? (Configurável: padrão = manter; admin pode remover manualmente)
- [ ] AC9: Logs em Sentry com tag `stripe_event_id` e `event_type`
- [ ] AC10: Teste manual com Stripe CLI: `stripe trigger checkout.session.completed`; replay do mesmo evento confirma idempotência (não duplica payment)

## Escopo

**IN:**
- Route Handler do webhook
- 4 event handlers
- Idempotência via `webhook_events`
- Cross extensions aplicadas no handler
- Logs Sentry

**OUT:**
- Auto renewal scheduling (F7.5)
- Cron de expiração (F7.5)
- Emails de cada caso (F7.6)

## Contexto Técnico

- Risco crítico (ADR-001): webhook falha → aluno paga sem acesso. Idempotência + endpoint manual de reprocessar (F6.4) + Sentry alerta
- Stripe envia evento múltiplas vezes; UNIQUE em `stripe_event_id` é a única defesa
- ADR-001 D14: cross extensions aplicadas no webhook, não no checkout
- Performance: webhook deve responder em < 5s; lógica pesada (geração de PDF, envio de email) idealmente em background, mas no MVP inline está ok
