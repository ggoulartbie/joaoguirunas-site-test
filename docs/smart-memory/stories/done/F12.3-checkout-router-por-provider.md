# Story F12.3: Router de checkout por provider (Stripe vs InfinitePay)

## Objetivo
Adaptar a server action de checkout público (`checkoutPublic.ts`) para ler `cohort.payment_provider` e despachar para o adapter correto (StripeAdapter ou InfinitePayAdapter), e ajustar a página de sucesso para funcionar com ambos os providers.

## Acceptance Criteria
- [x] AC1: `src/app/actions/checkoutPublic.ts` lê `cohort.payment_provider` após validar a cohort. Se `'INFINITEPAY'`, chama `InfinitePayAdapter.createCheckoutLink(...)` (de F12.1). Se `'STRIPE'` (default), mantém comportamento atual via `StripeAdapter.createCheckoutSession(...)`. Se valor inesperado (defesa em profundidade), retorna erro `'Provider de pagamento inválido.'` Em ambos os casos, persiste linha em `payments` com `payment_provider` correspondente antes do redirect (Stripe já fazia; InfinitePay precisa do `order_nsu` UUID conforme AC3 da F12.1).
- [x] AC2: Quando provider = `INFINITEPAY`, `redirect_url` enviado ao InfinitePay aponta para `${NEXT_PUBLIC_APP_URL}/academy/checkout/sucesso?provider=infinitepay&order_nsu={uuid}` (substituir `{uuid}` pelo UUID gerado e gravado em `payments.id`/`infinitepay_order_nsu`). Quando provider = `STRIPE`, mantém URL atual com `session_id={CHECKOUT_SESSION_ID}`. `cancel_url` aponta para `/curso-online` em ambos.
- [x] AC3: `src/app/(academy)/academy/checkout/sucesso/page.tsx` lê `searchParams.provider`. Se `'infinitepay'`, busca `payments` por `id = order_nsu` e exibe a confirmação (mesmo layout, dados de `payments.amount_cents`, `cohort_id`, `created_at`). Se `'stripe'` ou ausente (compatibilidade retroativa com Stripe sem o param), busca por `session_id` como hoje. Página não quebra se `order_nsu` for inválido — mostra mensagem "Pagamento em processamento, verifique seu email" e link para `/academy/login`.
- [x] AC4: `pnpm typecheck` limpo. Smoke manual cobrindo: (a) compra cohort com provider=STRIPE redireciona para Stripe e volta com `session_id` — fluxo legado intacto; (b) compra cohort com provider=INFINITEPAY redireciona para InfinitePay e volta com `?provider=infinitepay&order_nsu=...` — nova rota; (c) parâmetros faltantes ou inválidos não causam 500.

## Escopo

**IN:**
- `src/app/actions/checkoutPublic.ts` — adicionar branch por provider
- `src/app/(academy)/academy/checkout/sucesso/page.tsx` — suporte a ambos providers via query param
- Eventuais tipos (`PaymentProvider = 'STRIPE' | 'INFINITEPAY'`) em `src/lib/payment/types.ts` ou similar

**OUT:**
- Implementação do InfinitePayAdapter (F12.1)
- Migration da coluna `payment_provider` (F12.2)
- LP `/curso-online` continua sem mudança visual — provider é decidido server-side com base na cohort
- Refactor da página de sucesso (manter layout atual, apenas adicionar a branch de leitura)

## Contexto Técnico

- **`checkoutPublic.ts` atual:** valida `slug` via Zod, busca cohort, checa `is_purchasable` + `stripe_price_entry_id`, monta sessão via `StripeAdapter`, faz `redirect`. Auditoria recente em `[[F9.2-auditar-checkout-stripe-curso-online]]` documenta o fluxo.
- **`payments.id`** já é UUID — pode ser usado diretamente como `order_nsu` da F12.1 (consistência: mesma chave em ambos os fluxos). InfinitePay envia `order_nsu` de volta no webhook → fácil casar.
- **Página de sucesso atual:** lê `session_id` via `searchParams`, busca `payments` por `stripe_checkout_session_id`. Mantém esse caminho como default, adicionar branch novo para `provider=infinitepay`.
- **Validação extra recomendada (defesa em profundidade):** quando provider = INFINITEPAY, antes do redirect, validar que a cohort não tem `stripe_price_entry_id` populado SEM `payment_provider='STRIPE'` (cenário de inconsistência). Não bloqueante — pode logar e prosseguir.
- **Compatibilidade retroativa:** todos os links Stripe existentes (sucesso?session_id=...) continuam funcionando — branch default da página cobre.
- **Dependências:** F12.1 (adapter pronto) e F12.2 (coluna `payment_provider` existe) precisam estar mergeadas antes desta story ser implementada.

### Issues observacionais (não bloqueantes)

- **[CONCERN-1]** A página de sucesso busca `payment.status` mas não exibe `amount_cents`, `cohort_id` ou `created_at` mencionados no AC3 ("dados de payments.amount_cents, cohort_id, created_at"). A interpretação adotada foi mostrar layout idêntico ao Stripe (next steps + headings) em vez de detalhar valores — UX consistente com o Stripe success. **Aceitável** porque o layout fica unificado, mas se o produto quiser detalhamento (ex.: "Você pagou R$XX para curso Y"), seria fácil adicionar. Não bloqueante.

- **[CONCERN-2]** No fluxo InfinitePay, o email é **obrigatório** se não houver auth user (linha 91-93) — divergência sutil do Stripe, que aceita email opcional pois o checkout do Stripe captura. Comportamento documentado no comentário linhas 50-52 e essencial para `findOrCreateUser` antes do INSERT em `payments`. UX OK desde que o form de checkout exija email para cohorts INFINITEPAY (verificar com sites-dev-alpha em F9.18 follow-up).

- **[CONCERN-3]** `fireCrmWebhook` é disparado mesmo quando o adapter de pagamento falha depois (linha 104, antes de `createCheckoutLink`). Isso significa CRM recebe lead mesmo sem pagamento — comportamento aceitável (lead capture é a intenção mesmo se pagamento falhar), mas vale documentar.

- **[CONCERN-4]** Linhas 127-128 — `new InfinitePayAdapter()` em vez de usar o `infinitePay` Proxy singleton exportado em `infinitepay.ts:101`. Funcional, mas perde o benefício do lazy-init singleton (cria adapter novo a cada request). Pequena regressão de performance, não-bloqueante. Sugestão: usar `infinitePay.createCheckoutLink(...)` direto.

- **[CONCERN-5]** Validação extra recomendada do contexto técnico (linha 49 da story) — "validar que a cohort não tem `stripe_price_entry_id` populado SEM `payment_provider='STRIPE'`" — **não foi implementada**. Era nice-to-have explicitamente não-bloqueante. Aceito.
