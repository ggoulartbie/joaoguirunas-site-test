# Story F13.2: Excluir pagamentos pendentes no admin

## Objetivo
Permitir que o admin exclua linhas de pagamento com status `PENDING` na tabela em `/academy/admin/pagamentos`, com confirmação simples e guard server-side que rejeita qualquer status diferente de `PENDING`.

## Acceptance Criteria

- [ ] **AC1** — Botão "Excluir" visível apenas em linhas com `status === 'PENDING'` na tabela renderizada por `PaymentsClient.tsx`. Linhas com status `APPROVED`, `DECLINED`, `REFUNDED` (ou outros) NÃO mostram o botão. Estilo destrutivo (vermelho), ícone `Trash2`.
- [ ] **AC2** — Ao clicar em "Excluir", abre modal de confirmação simples (sem input de digitação): texto "Tem certeza que deseja excluir este pagamento pendente? Esta ação não pode ser desfeita." + botões "Cancelar" e "Excluir".
- [ ] **AC3** — Server action `deletePayment(paymentId: string)` em `src/app/(academy)/academy/(admin)/admin/pagamentos/actions.ts`:
  - Valida sessão admin (RBAC) antes de executar.
  - Busca o payment por id em `payments`.
  - Guard obrigatório: se `status !== 'PENDING'`, retorna `{ error: 'Apenas pagamentos pendentes podem ser excluídos.' }` (equivalente a 400).
  - Se `status === 'PENDING'`, executa `delete from payments where id = $1 and status = 'PENDING'` (guard duplo em SQL para evitar TOCTOU).
  - Retorna `{ success: true }` ou `{ error: string }`.
- [ ] **AC4** — Após sucesso: modal fecha, linha desaparece da tabela (via `router.refresh()`), toast verde "Pagamento excluído com sucesso".
- [ ] **AC5** — Botão NÃO aparece em linhas com status diferente de `PENDING`. Server action rejeita mesmo se chamada diretamente com paymentId de linha não-PENDING (defesa em profundidade contra request manual).

## Escopo

**IN:**
- Botão "Excluir" condicional (apenas em linhas PENDING) em `PaymentsClient.tsx`.
- Modal de confirmação simples.
- Server action `deletePayment(paymentId)` com guard duplo (TypeScript + SQL).
- Refresh da tabela após delete.
- Tratamento de erro com toast.

**OUT:**
- Exclusão em lote.
- Excluir pagamentos APPROVED/DECLINED/REFUNDED (fora de escopo — histórico fiscal).
- Cancelar checkout session no Stripe/InfinitePay (a sessão expira sozinha; delete apenas remove a linha local).
- Audit log persistente.

## Contexto Técnico

**Arquivos afetados:**
- `src/app/(academy)/academy/(admin)/admin/pagamentos/actions.ts` — adicionar `deletePayment` action
- `src/app/(academy)/academy/(admin)/admin/pagamentos/PaymentsClient.tsx` — botão condicional + modal de confirmação

**Dependências:**
- Tabela `payments` com coluna `status` (enum: PENDING, APPROVED, DECLINED, REFUNDED).
- `supabaseAdmin` para bypass RLS na operação.
- RBAC admin já validado pelo middleware/`(admin)` route group.

**Constraints:**
- Guard duplo (TS + SQL WHERE) — não confiar apenas no filtro UI.
- Idempotência: se a linha já não existir, retornar `{ success: true }`.
- Pagamentos PENDING tipicamente não têm `stripe_payment_intent_id` ou `infinitepay_order_nsu` preenchidos (apenas `stripe_checkout_session_id` ou `infinitepay_order_nsu` como referência). Delete não impacta provider externo — sessão expira naturalmente.

**Decisões:**
- Confirmação simples (sem digitar palavra) — risco baixo, dado é descartável (PENDING significa que o pagamento nunca foi confirmado).
- Não cancelar sessão remota — economia de chamada externa e sessão expira por TTL.
