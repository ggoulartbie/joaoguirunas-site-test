# Story F9.9: Auditar has_access RPC + gates de acesso a aulas e módulos

## Objetivo
Garantir que o RPC `has_access(p_user_id, p_lesson_id)` bloqueia corretamente o acesso a aulas de turmas não compradas, módulos não incluídos na cohort do aluno e matrículas expiradas — protegendo `video_id`, conteúdo e materiais.

## Acceptance Criteria
- [ ] AC1: Aluno **não matriculado** acessando `/academy/curso/[slug]/aula/[lesson-slug]` recebe `hasAccess=false` e vê o `LockedContent` em vez do `VideoPlayer`. O `lesson.video_id` **nunca é exposto** no HTML quando `hasAccess=false` (conferir via curl + view source).
- [ ] AC2: Aluno matriculado em cohort `X` que tem `cohort_courses.included_module_ids` listando apenas módulos `[A, B]` recebe `hasAccess=false` para aulas do módulo `C`. Página `/academy/curso/[slug]` mostra módulo `C` como `LockedModuleCard`.
- [ ] AC3: Membership com `status='EXPIRED'` retorna `hasAccess=false` mesmo que `cohort_courses` liste o módulo. Membership com `expires_at` no passado mas ainda `status='ACTIVE'` (aguardando cron) também retorna `hasAccess=false` (RPC checa `expires_at > now()` ou apenas `status`).
- [ ] AC4: Usuário com `role='ADMIN'` ou `role='MENTOR'` recebe `hasAccess=true` para qualquer aula (override de role). Documentar este comportamento em comentário do RPC.
- [ ] AC5: O fallback `hasGlobalAccess = (cohortCourseRows ?? []).length === 0` em `src/app/(academy)/academy/(student)/curso/[slug]/page.tsx` (linha 143) e similar em `dashboard/page.tsx` é seguro: validar que **só libera quando o aluno tem cohort ativa mas a tabela `cohort_courses` está vazia para essa cohort** (i.e., cohort sem restrições configuradas, não cohort inexistente). Caso suspeito: aluno sem nenhuma cohort não deve receber `hasGlobalAccess=true`.

## Escopo

**IN:**
- Função SQL `has_access(p_user_id, p_lesson_id)` em `supabase/migrations/`
- Auditoria das 4 chamadas existentes a `has_access` (grep no repo)
- Lógica de `hasGlobalAccess` em `curso/[slug]/page.tsx` e `dashboard/page.tsx`
- Teste manual com 4 perfis: anon, aluno-sem-cohort, aluno-cohort-restrita, admin

**OUT:**
- Reescrever RLS de tabelas (escopo F1.5)
- Mudança no schema de `cohort_courses` ou `cohort_members`
- Implementação de SSO ou novos providers

## Contexto Técnico
- O RPC `has_access` é a barreira final. Qualquer bug aqui = vazamento de vídeo/conteúdo pago.
- Em `aula/[lesson-slug]/page.tsx` (linha 83-88), o RPC é chamado **antes** de renderizar o VideoPlayer, e `if (!hasAccess) return <LockedContent />` (linha 194).
- Cuidado: `supabaseAdmin` é usado para queries de navegação (sidebarModules, allLessons) — isso bypassa RLS mas não expõe `video_id` se o gate principal está correto.
- `hasGlobalAccess` é um fallback usado na UI para não bloquear admins / casos sem cohort_courses populado, mas merece validação rigorosa.
- Em `curso/[slug]/page.tsx` linha 138-143 a lógica diz "If cohort_courses is empty, assume full access (direct enrollment or admin)" — confirmar se é intencional.

### Issues observacionais

- **[CONCERN-1]** O override ADMIN/MENTOR em AC4 é amplo: ADMIN/MENTOR têm `has_access=true` para QUALQUER aula (independente de course/cohort). Comportamento intencional e documentado, mas amplifica o blast radius se uma conta admin/mentor for comprometida. **Sugestão hardening (não-bloqueante):** considerar restrição por curso (`MENTOR` só pode acessar cursos que mentora) numa story futura. Não afeta segurança da feature atual.
- **[CONCERN-2]** `hasGlobalAccess` é mantido na UI mesmo após o RPC ter override de ADMIN — é redundância segura (defense in depth) e mantém compatibilidade com legado. Aceito.
