# Story F6.3: Admin — CRUD de cohorts (coração comercial)

## Objetivo

Interface completa de criação/edição de cohort com 9 seções (Identidade, Status, Acesso, Cursos liberados, Comercial, Cross extensions, Membros, Sessões, Cupons), conforme PRD Seção 6.3.

## Acceptance Criteria

- [ ] AC1: `/admin/turmas` lista cohorts com filtros (status, is_purchasable, ocupação) + botão "Nova turma"
- [ ] AC2: `/admin/turmas/[id]/editar` com 9 seções em tabs ou wizard:
  1. **Identidade**: name, slug, description, cover_image_url
  2. **Status e datas**: status, start_date, end_date, total_seats
  3. **Acesso**: access_duration_days (ou vitalício), group_url, has_live_sessions, has_support
  4. **Cursos liberados**: checklist hierárquico (course → modules); grava em `cohort_courses.included_module_ids`
  5. **Comercial**: is_purchasable, has_public_page, entry_price_cents, extension_price_cents, max_installments_*, extension_duration_days, allows_auto_renewal
  6. **Extensões cruzadas**: matriz CRUD de `cohort_cross_extensions` onde source = cohort atual; selector de target + days_granted + descrição
  7. **Membros**: lista de cohort_members com busca, ações (adicionar, remover, alterar prazo, ver progresso)
  8. **Sessões**: CRUD de live_sessions
  9. **Cupons**: CRUD de coupons da cohort
- [ ] AC3: Salvar cohort com `is_purchasable=true` dispara sync com Stripe (Fase 7); por enquanto stub que registra em log
- [ ] AC4: Validação Zod por seção; erros isolados por tab
- [ ] AC5: Preço em centavos no banco mas input em reais na UI (helper `formatBRL` / `parseBRL`)
- [ ] AC6: Tab "Membros": busca por nome/email, adicionar manualmente cria `cohort_members` ACTIVE com expires_at customizável
- [ ] AC7: Indicador de "Vagas restantes" calculado (`total_seats - filled_seats`)

## Escopo

**IN:**
- Página de listagem
- Página de edição com 9 seções
- Lógica completa de cohort_courses, cross_extensions, cohort_members manuais
- Sync stub com Stripe (real na F7.2)

**OUT:**
- Sync Stripe efetivo (F7.2)
- Webhook handlers (F7.3)
- Live session com integração Meet/Zoom API (post-MVP)

## Contexto Técnico

- Cohort é o **coração** do admin — tempo investido aqui paga retorno em tudo
- Mudanças em `cohort_courses` ou `included_module_ids` afetam quem tem acesso (via `has_access`) — invalidar caches relevantes
- Mudança de preço: NÃO altera membros existentes; só novos checkouts
