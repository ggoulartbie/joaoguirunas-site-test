# Story F7.2: Sincronização Cohort ↔ Stripe

## Objetivo

Quando cohort é salva como `is_purchasable=true`, criar/atualizar Stripe Product e 2 Prices (entry e extension); idempotente. Mesma lógica para cupons.

## Acceptance Criteria

- [ ] AC1: Função `syncCohortWithStripe(cohortId)` em `lib/payment/sync.ts`:
  - Se cohort `is_purchasable=true` e sem `stripe_price_entry_id`: cria Stripe Product + 2 Prices, salva IDs
  - Se cohort `is_purchasable=true` com IDs existentes: verifica drift (preços/parcelas diferentes), arquiva price antigo e cria novo se necessário
  - Se cohort `is_purchasable=false`: arquiva todos os prices ativos
- [ ] AC2: Disparada após `saveCohort` Server Action (F6.3)
- [ ] AC3: Idempotente: rodar 2x consecutivas sem mudanças não cria nada
- [ ] AC4: `syncCouponWithStripe(couponId)` análoga: cria Stripe Coupon e linka via `coupons.stripe_coupon_id`
- [ ] AC5: Logs estruturados em cada operação (Sentry)
- [ ] AC6: Endpoint admin `/admin/turmas/[id]/resync-stripe` para forçar sync manual
- [ ] AC7: Testes manuais: editar preço de cohort dispara update no Stripe; arquivar cohort arquiva price

## Escopo

**IN:**
- Função sync para cohorts e coupons
- Trigger pós-save
- Endpoint manual de resync

**OUT:**
- Webhook (F7.3)
- Checkout em si (F7.4)

## Contexto Técnico

- Stripe Prices são imutáveis em `unit_amount` — mudar preço requer arquivar antigo + criar novo
- Risco (ADR-001): sincronização fora de sync — mitigação é a função idempotente + endpoint manual
- Stripe Subscription Schedule (auto renewal) usa o `price` ativo no momento da cobrança
