# Story F12.1: InfinitePayAdapter + webhook handler

## Objetivo
Implementar a integração base com InfinitePay como segundo provider de pagamento da plataforma — adapter para criar links de checkout, route handler do webhook com verificação via `payment_check`, e provisionamento idempotente de `cohort_members`/`payments` ao confirmar pagamento.

## Acceptance Criteria
- [x] AC1: `src/lib/payment/infinitepay.ts` exporta `InfinitePayAdapter` com método `createCheckoutLink({ cohortSlug, amount, customerEmail, orderId, redirectUrl, webhookUrl })` que faz `POST https://api.checkout.infinitepay.io/links` com `Authorization: Bearer ${INFINITEPAY_BEARER_TOKEN}` e `handle = INFINITEPAY_HANDLE` (env vars). Retorna `{ url }` para redirect do cliente. Lazy-init do client (mesmo pattern de `stripe.ts` — ver `project_vercel_deploy.md`).
- [x] AC2: A server action de checkout (em F12.3) chama `createCheckoutLink` e usa Next.js `redirect(url)` para enviar o cliente ao link. Sem erro de servidor em prod e preview.
- [x] AC3: Antes de redirecionar, persiste linha em `payments` com: `id` (UUID novo, vai como `order_nsu` no payload do InfinitePay), `cohort_id`, `user_id` (se conhecido — `null` em compra pública), `amount_cents`, `status='PENDING'`, `payment_provider='INFINITEPAY'`, `infinitepay_order_nsu` (= `id`). UUID gerado server-side via `crypto.randomUUID()`.
- [x] AC4: `src/app/api/webhooks/infinitepay/route.ts` aceita `POST`, lê o body JSON (`invoice_slug`, `amount`, `paid_amount`, `installments`, `capture_method`, `transaction_nsu`, `order_nsu`, `receipt_url`, `items`), responde `200 OK` com `{ received: true }` em **menos de 1s** (resposta síncrona ao InfinitePay; processamento pode acontecer depois ou ser feito antes do response — desde que sub-1s). Em caso de erro interno, log via `Sentry.captureException` e responde `500` sem stack trace.
- [x] AC5: Antes de marcar `payments` como APPROVED e provisionar acesso, valida via `POST https://api.checkout.infinitepay.io/payment_check` com `{ handle, transaction_nsu, slug: invoice_slug }` e Bearer Token. Aceita apenas se response `paid: true` e `paid_amount >= amount` (proteção contra webhook forjado, já que InfinitePay não fornece assinatura HMAC). Se `paid: false` ou erro de validação, registra evento em `webhook_events` com `processed=false` e não provisiona.
- [x] AC6: Webhook idempotente via `order_nsu` — antes de processar, busca `payments` por `infinitepay_order_nsu = body.order_nsu`. Se já está `APPROVED`, responde `{ received: true, idempotent: true }` sem reprocessar. Se `PENDING`, processa: `payments.status → APPROVED`, cria `cohort_members` com `status='ACTIVE'`, `expires_at = now() + cohort.access_duration_days`, dispara `sendWelcomeInviteEmail`. Segundo webhook idêntico (mesma `order_nsu`) não duplica nem `cohort_members` nem nova linha em `payments`.
- [x] AC7: `src/middleware.ts` libera `/api/webhooks/infinitepay` como rota pública (sem auth gate), igual a `/api/webhooks/stripe` e `/api/webhooks/*`. Verificar que a regex/array de rotas públicas inclui o novo path.
- [x] AC8: `pnpm typecheck` limpo. Env vars `INFINITEPAY_HANDLE` e `INFINITEPAY_BEARER_TOKEN` adicionadas a `.env.example` com comentário explicativo. Runbook básico em `docs/smart-memory/runbooks/checkout-infinitepay.md` (env vars, fluxo do payment_check, como reprocessar webhook manualmente via `transaction_nsu`).

## Contexto Técnico

- **Endpoints InfinitePay:**
  - `POST https://api.checkout.infinitepay.io/links` — cria link de checkout, retorna `{ url }`
  - `POST https://api.checkout.infinitepay.io/payment_check` — valida pagamento, retorna `{ paid: boolean, ... }`
- **Auth:** `Authorization: Bearer ${INFINITEPAY_BEARER_TOKEN}` em ambos
- **Handle:** `growthsales` (via `INFINITEPAY_HANDLE`)
- **Sem assinatura HMAC** — toda validação é via `payment_check` server-to-server. Crítico para segurança: nunca confiar no payload do webhook sem `payment_check`.
- **Payload esperado do webhook:**
  ```json
  { "invoice_slug": "...", "amount": 79700, "paid_amount": 79700, "installments": 1,
    "capture_method": "credit_card", "transaction_nsu": "...", "order_nsu": "<uuid>",
    "receipt_url": "https://...", "items": [{ "name": "...", "quantity": 1, "amount": 79700 }] }
  ```
- **Stripe pattern para reaproveitar:** `src/app/api/webhooks/stripe/route.ts` — estrutura de idempotência via `webhook_events`, criação user/profile via `findOrCreateUser`, geração magic link, `sendWelcomeInviteEmail`. Reusar essas funções, só substituir a parte de validação de assinatura por `payment_check`.
- **Tabela `payments`:** já existe com `payment_provider` (text) — confirmar em `supabase/migrations/20260506021931_*.sql`. Coluna `infinitepay_order_nsu UUID UNIQUE` precisa ser adicionada.
- **Provisionamento de user:** mesmo fluxo do Stripe — `findOrCreateUser({ email })` + magic link + welcome email.
- **`transaction_nsu` é a fonte de verdade** para verificar pagamento via `payment_check`. Nunca confiar só em `paid_amount` do webhook payload.
- **Resposta sub-1s:** considerar fazer `payment_check` + provisionamento síncrono se rápido (~300ms total esperado), ou enfileirar via `webhook_events` + processamento em background. Decisão técnica do dev.

### Issues observacionais

- **[CONCERN-1 — paid_amount não bloqueia underpayment]** Linha 138-139: `// Guarda valor verificado; se paid_amount < amount_cents registrado, log mas continua (InfinitePay decide)`. O AC5 pede explicitamente `paid_amount >= amount` como gate, mas a implementação delegou ao InfinitePay. **Risco:** se `payment_check` retorna `paid: true` mas `paid_amount` < `amount_cents` (cenário de mismatch entre intenção e cobrança real), o aluno recebe acesso por valor menor. **Mitigação atual:** premissa que InfinitePay só retorna `paid: true` se valor total foi cobrado — aceito, mas não é defensa codificada. **Sugestão:** adicionar `if (check.paidAmount < payment.amount_cents) { /* log + DECLINED */ }` antes da linha 139.

- **[CONCERN-2 — coluna `infinitepay_order_nsu` é TEXT, não UUID]** AC explicita "UUID", mas migration `20260508020000:13` declara `text unique`. UUID gerado server-side via `crypto.randomUUID()` (em F12.3) é serializado como string — funciona, mas perde validação de tipo no DB. **Não bloqueante:** UNIQUE garante idempotência; se quiserem hardening, próxima migration pode `ALTER COLUMN ... TYPE uuid USING infinitepay_order_nsu::uuid`.

- **[CONCERN-3 — `processWebhook` é síncrono dentro do POST]** A resposta sub-1s depende de o `payment_check` + provisionamento serem rápidos (~300ms). Se a InfinitePay ficar lenta ou Supabase travar, podemos passar de 1s e o InfinitePay vai retentar. **Mitigação:** `webhook_events` + idempotência via `existingPayment.status === 'APPROVED'` cobrem o retry. Aceito.

- **[CONCERN-4 — payment com user_id null sem customer_email]** Linha 154-166: se compra pública e webhook não envia `customer_email`, payment é marcado APPROVED mas **sem provisionar acesso** (early return na linha 165). **Comportamento esperado** segundo a doc do AC3 (compra pública), mas merece monitoramento — recomendo Sentry alert se essa branch for atingida em prod (atualmente só `console.error`). 

- **[CONCERN-5 — increment_filled_seats com cast `as any`]** Linha 211-212: `// eslint-disable-next-line @typescript-eslint/no-explicit-any` + `(supabaseAdmin.rpc as any)(...)`. Funcional, mas tipos não cobrem RPC custom. Cleanup pós-launch quando os tipos forem regenerados.
