# Story F1.3: Migration — Cohorts (unidade central)

## Objetivo

Criar migration com as 5 tabelas de cohorts (`cohorts`, `cohort_courses`, `cohort_cross_extensions`, `cohort_members`, `live_sessions`) — o coração comercial e de acesso da plataforma.

## Acceptance Criteria

- [x] AC1: Tabela `cohorts` com todos os campos da PRD Seção 5.2 (identidade, status, datas, vagas, acesso, comercial, Stripe)
- [x] AC2: `cohort_courses` com PK composta (cohort_id, course_id) e coluna `included_module_ids uuid[]`
- [x] AC3: `cohort_cross_extensions` com unique (source_cohort_id, target_cohort_id), `days_granted`, `is_active`, `description`
- [x] AC4: `cohort_members` com unique (cohort_id, user_id), `member_role`, `status`, `expires_at`, `auto_renew_enabled`, `next_renewal_at`
- [x] AC5: `live_sessions` com FK opcional para `lessons.id`
- [x] AC6: Triggers de `updated_at` aplicados onde aplicável
- [x] AC7: Índices em `cohort_members(user_id)`, `cohort_members(status, expires_at)` para queries de `has_access`
- [x] AC8: `supabase db push` aplica limpo e tipos TS regenerados
