# Story 2.2: Admin gerencia materiais da aula (listar + excluir)

## Objetivo
Garantir que no editor de aula admin (`/academy/admin/cursos/[courseId]/aulas/[lessonId]`) o admin consiga **ver todos os materiais já anexados** (arquivos E links) e **excluí-los individualmente** com cleanup do arquivo no Supabase Storage e confirmação destrutiva — eliminando o estado "subi um teste e não tenho como remover".

## Acceptance Criteria

- [ ] **AC1 (UI — listar)**: A seção "Materiais" do `LessonEditorClient` lista 100% dos materiais da aula independente do `kind`. Materiais `LINK` (que têm `storage_path = null`) também aparecem na listagem com indicador visual de tipo "LINK" e o `external_url` truncado/encurtado.
- [ ] **AC2 (UI — excluir)**: Cada item da lista tem botão de exclusão visível. O botão **aparece para LINK também** (hoje só aparece se `storage_path` for truthy — ver `LessonEditorClient.tsx:306`). Clique abre confirmação destrutiva (`confirm()` nativo é aceitável; modal estilizado é nice-to-have).
- [ ] **AC3 (Server action — hard delete)**: A action `deleteMaterial(materialId, storagePath, courseId)` em `src/app/(academy)/academy/(admin)/admin/cursos/actions.ts:211` deve:
  - (a) tolerar `storagePath` nulo/vazio (caso LINK) sem chamar `storage.remove([''])` — pular o `storage.from('materials').remove(...)` quando vazio.
  - (b) executar hard delete na tabela `materials` (não há `deleted_at` na tabela; soft delete está OUT).
  - (c) revalidar `/academy/admin/cursos/[courseId]` E `/academy/admin/cursos/[courseId]/aulas/[lessonId]` para refletir a mudança imediatamente no editor de aula.
- [ ] **AC4 (Optimistic + rollback)**: O handler `handleDeleteMaterial` no client deve:
  - (a) `await` a server action dentro do `try/catch` (atualmente é fire-and-forget em `startTransition` sem catch — `LessonEditorClient.tsx:128`).
  - (b) Em caso de erro do server: **rollback** do state local (re-inserir o material removido) e exibir mensagem de erro via `setError`.
  - (c) Em caso de sucesso: confirmar a remoção via `router.refresh()` para sincronizar com o servidor.
- [ ] **AC5 (Upload sync)**: Após `uploadMaterial` em `handleFileUpload` (`LessonEditorClient.tsx:108-123`), o material recém-enviado aparece na listagem **sem precisar recarregar manualmente** a página. Solução aceita: `router.refresh()` já está lá (linha 116) — validar que o `initialMaterials` é re-fetched pelo server component pai. Se necessário, o pai precisa ser server component que fetch materiais a cada navegação.
- [ ] **AC6 (Storage cleanup)**: Para materiais `kind != 'LINK'` com `storage_path` válido, o arquivo no bucket privado `materials/lessons/{lessonId}/{uuid}.{ext}` é removido junto com a row. Para `LINK`, apenas a row é removida.
- [ ] **AC7 (RLS defesa em profundidade)**: Confirmar que a policy `"materials: admin deleta"` em `supabase/migrations/20260506022037_has_access_rls_policies.sql:190` cobre o DELETE. Action usa `supabaseAdmin` (bypassa RLS), mas a policy continua sendo defesa em profundidade. Não criar nova migration — apenas validar no checklist.
- [ ] **AC8 (Smoke manual)**: Como admin, em aula de teste: (1) subir 1 PDF, ver aparecer na lista; (2) subir 1 link via cara complementar (se in-scope), ver aparecer; (3) excluir o PDF — desaparece da lista E da página do aluno (não conta mais como material disponível); (4) excluir um LINK — desaparece da lista (sem erro).

## Escopo

**IN:**
- **UI (sites-dev-alpha)**: editar `src/app/(academy)/academy/(admin)/admin/cursos/[courseId]/aulas/[lessonId]/LessonEditorClient.tsx`:
  - Remover guard `mat.storage_path && ...` na linha 306 — botão delete aparece para todos os kinds.
  - Para `kind === 'LINK'`, exibir o `external_url` truncado abaixo do título (substitui o `formatBytes` que não se aplica).
  - Refatorar `handleDeleteMaterial` para await + try/catch + rollback (AC4).
  - Confirmar/melhorar feedback visual de loading no item sendo deletado (estado `deleting` por id).
- **Server action (sites-dev-beta)**: editar `src/app/(academy)/academy/(admin)/admin/cursos/actions.ts:211-217`:
  - Tolerar `storagePath` vazio/null antes de chamar `storage.remove` (AC3a).
  - Adicionar `revalidatePath('/academy/admin/cursos/[courseId]/aulas/[lessonId]', 'page')` além do path do curso (AC3c).
  - Logar (não throw) erro do `storage.remove` para não bloquear o DELETE da row caso o arquivo já não exista no bucket.

**OUT:**
- Soft delete (tabela não tem `deleted_at`).
- Audit log de deletes (sem requisito).
- Restringir delete por `uploaded_by` — coluna não existe na tabela `materials` e não há requisito de ownership; qualquer admin pode deletar.
- Bulk delete (deletar múltiplos de uma vez).
- Modal de confirmação estilizado custom (nice-to-have; `confirm()` nativo já está em uso e é aceitável).
- Adicionar ao editor o fluxo "adicionar link externo" (`addLinkMaterialAction` existe em `material-actions.ts:158` mas a UI no editor só tem upload de arquivo — pode ficar para follow-up).
- Reordenação por drag-and-drop dos materiais.
- Editar título de material existente.
- Migration nova (RLS está OK; storage policy está OK).
- Touch no `MaterialsUpload.tsx` standalone (componente não está sendo usado neste editor — o editor tem código próprio).

## Contexto Técnico

**Arquivos afetados:**
- `src/app/(academy)/academy/(admin)/admin/cursos/[courseId]/aulas/[lessonId]/LessonEditorClient.tsx` (linhas 64, 108-129, 293-314)
- `src/app/(academy)/academy/(admin)/admin/cursos/actions.ts` (linhas 211-217)

**Diagnóstico:**

1. **Botão delete não aparece para LINK** (`LessonEditorClient.tsx:306`):
   ```tsx
   onClick={() => mat.storage_path && handleDeleteMaterial(mat.id, mat.storage_path)}
   ```
   Para `kind === 'LINK'`, `storage_path` é `null` e o botão fica inerte. Admin que adicionou link (via `addLinkMaterialAction`) fica preso.

2. **Optimistic sem rollback** (`LessonEditorClient.tsx:125-129`):
   ```tsx
   async function handleDeleteMaterial(id: string, storagePath: string) {
     if (!confirm('Remover material?')) return
     setMaterials((prev) => prev.filter((m) => m.id !== id))
     startTransition(() => deleteMaterial(id, storagePath, courseId))
   }
   ```
   Se a server action falhar, o item desaparece da UI mas continua no banco — admin não percebe. `startTransition` sem `await`/`catch` engole o erro.

3. **Server action sem guard de storage_path vazio** (`actions.ts:213`):
   ```ts
   await supabaseAdmin.storage.from('materials').remove([storagePath])
   ```
   Para LINK, `storagePath` chega como `null` ou `''`. O Supabase ainda processa (no-op ou erro silencioso), mas é limpo trocar por `if (storagePath) { ... }`.

4. **Revalidação parcial** (`actions.ts:208, 216`):
   Hoje revalida `/academy/admin/cursos/${courseId}` mas não a rota da aula em si — após delete/upload no editor de aula, dependemos só do `router.refresh()` do client. Adicionar revalidação explícita do path da aula é defesa.

**Schema relevante (`supabase/migrations/20260506021755_initial_schema_identidade_catalogo.sql:131-143`):**
```sql
create table public.materials (
  id            uuid primary key default gen_random_uuid(),
  lesson_id     uuid not null references public.lessons(id) on delete cascade,
  title         text not null,
  kind          text not null check (kind in ('PDF', 'ZIP', 'IMAGE', 'LINK', 'OTHER')),
  storage_path  text,         -- nullable; LINK não tem
  external_url  text,         -- nullable; só LINK tem
  size_bytes    bigint,
  sort_order    int not null default 0,
  created_at    timestamptz not null default now()
);
-- materials não tem updated_at: imutável após upload
-- não tem deleted_at: hard delete
-- não tem uploaded_by: sem ownership
```

**RLS (`20260506022037_has_access_rls_policies.sql:190`):** policy `"materials: admin deleta"` — cobre DELETE para role admin (verificar via `is_admin()` helper).

**Storage path convention** (`actions.ts:189`): `lessons/{lessonId}/{uuid}.{ext}` no bucket privado `materials`.

**Coordenação:**
- sites-dev-alpha pega a parte UI (LessonEditorClient.tsx).
- sites-dev-beta pega a parte server action (actions.ts).
- Podem trabalhar em paralelo no mesmo branch (PR único). Beta entrega primeiro a action ajustada (interfaces estáveis) e Alpha encaixa o try/catch.
