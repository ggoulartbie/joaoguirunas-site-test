# Story F1.5: Função has_access() + RLS em todas as tabelas

## Objetivo

Implementar a função `has_access(user_id, lesson_id)` e habilitar RLS com policies explícitas em todas as tabelas.

## Acceptance Criteria

- [x] AC1: Função `public.has_access(p_user_id uuid, p_lesson_id uuid)` criada como `language sql security definer stable`
- [x] AC2: Função respeita `cardinality(included_module_ids) = 0` (vazio = todos) vs lista específica
- [x] AC3: Função verifica `cm.status = 'ACTIVE'` E `(cm.expires_at IS NULL OR cm.expires_at > now())`
- [x] AC4: RLS habilitado em todas as 21 tabelas do schema
- [x] AC5: Policies de `profiles`: SELECT/UPDATE próprio; ADMIN tudo (com WITH CHECK anti-escalada de role)
- [x] AC6: Policies de `courses`/`modules`/`lessons`: SELECT se published=true; ADMIN escreve
- [x] AC7: Policies de `materials`: SELECT se `has_access()` da lesson relacionada
- [x] AC8: Policies de `cohort_members`: usuário SELECT onde `user_id = auth.uid()`; ADMIN SELECT all; INSERT/UPDATE via service_role
- [x] AC9: Policies de `lesson_progress`: usuário CRUD próprias linhas (com WITH CHECK); ADMIN SELECT all
- [x] AC10: Policies de `comments`: SELECT se `has_access()`; INSERT autenticado com acesso; UPDATE/DELETE pelo autor com WITH CHECK
- [x] AC11: Policies de `forum_*`: leitura por qualquer cohort_member ACTIVE
- [x] AC12: Policies de `payments`: usuário SELECT próprios; service_role escreve
- [x] AC13: `webhook_events`: apenas service_role lê/escreve
- [x] AC14: Testes manuais cobrem: lesson liberada, lesson bloqueada, material, payment próprio/alheio
- [x] AC15: Documentação em `docs/smart-memory/agents/data-engineer/rls-policies.md`
