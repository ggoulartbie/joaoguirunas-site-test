# Story F9.13: Auditar Server Actions do aluno + integridade RLS

## Objetivo
Garantir que toda Server Action sob `(student)/` (a) chama `requireUser()` na primeira linha, (b) valida payload com Zod, (c) não usa `supabaseAdmin` (service role) para escrever em nome do usuário sem checagem explícita de ownership, e (d) que as RLS policies do Supabase não permitem leak cross-user mesmo sem `requireUser()` no caminho.

## Acceptance Criteria
- [ ] AC1: Inventário completo — listar toda Server Action em `src/app/(academy)/academy/(student)/**/actions.ts` e qualquer `*-actions.ts` (forum, materials, comments, progress, perfil). Adicionar à doc `docs/smart-memory/agents/research/student-actions-inventory.md`.
- [ ] AC2: Para cada action, confirmar:
  - Primeira chamada efetiva é `requireUser()` ou helper equivalente (`requireActiveMember()`).
  - Payload validado com `z.object({...}).safeParse(...)` antes de tocar Supabase.
  - Quando usa `supabaseAdmin`, o `user_id` no INSERT/UPDATE vem de `requireUser()`, NUNCA do payload (anti-IDOR).
  - Não vaza mensagem do Postgres para o cliente em caso de erro (sanitizar para `{ error: 'string genérico' }`).
- [ ] AC3: Smoke test de RLS — abrir Supabase SQL Editor com `auth.uid()` simulado de UserA e tentar SELECT/UPDATE em rows de UserB para tabelas: `lesson_progress`, `comments`, `forum_threads`, `forum_replies`, `votes`, `notifications`, `cohort_members`, `profiles`. Documentar resultados.
- [ ] AC4: `markAsAccepted` em `forum/actions.ts` valida ownership da thread (ok hoje) — verificar análogos em outras actions: editar/deletar comment próprio, editar perfil próprio, marcar lesson concluída só do próprio user.
- [ ] AC5: Server Action `requireActiveMember()` (forum) está duplicada conceitualmente em outras actions? Centralizar em `src/lib/auth/helpers.ts` se sim.
- [ ] AC6: Listar bugs encontrados como `S-NN — P{0-3}` em `docs/smart-memory/agents/research/student-actions-audit.md`. P0/P1 CORRIGIDOS na story; P2+ documentados.

## Escopo

**IN:**
- Todas as Server Actions sob `(academy)/(student)/`
- `material-actions.ts` em `(student)/curso/[slug]/aula/[lesson-slug]/`
- Smoke test de RLS via SQL Editor (anonymous + authenticated)
- Centralização de helpers de auth se houver duplicação

**OUT:**
- Server Actions admin (já auditadas em F9.4)
- Adicionar features novas
- Refactor de RLS policies (apenas verificar — fixes mais profundos viram nova story)
- API routes (auditadas em F9.5 e F9.10)

## Contexto Técnico
- `supabaseAdmin` em `src/lib/supabase/admin.ts` — service role key, bypassa RLS. Uso correto: SEMPRE atrás de `requireUser()` + ownership check.
- `requireUser()` e `requireAdmin()` em `src/lib/auth/helpers.ts`.
- `requireActiveMember()` aparece em `forum/actions.ts:9` — verificar duplicação.
- Padrão de erro do projeto: `{ error: string }` retornado pela action; nunca `throw` para o cliente.
- RLS policies definidas em F1.5 — referência: `[[F1.5-has-access-rls-policies]]`.
