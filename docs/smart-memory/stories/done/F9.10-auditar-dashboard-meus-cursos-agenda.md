# Story F9.10: Auditar dashboard + meus-cursos + agenda do aluno (queries cross-cohort, expiração, livestreams)

## Objetivo
Garantir que as três páginas de "home" do aluno (`/academy/dashboard`, `/academy/meus-cursos`, `/academy/agenda`) mostram dados corretos por cohort, calculam progresso fielmente, e expõem URL de live session apenas dentro da janela de 30min antes / 2h depois.

## Acceptance Criteria
- [ ] AC1: `/academy/dashboard` mostra "Continue de onde parou" apontando para a última aula com progresso > 0 e `completed=false`, com link válido `/academy/curso/{courseSlug}/aula/{lessonSlug}`. Aluno sem nenhum progresso vê fallback "Nenhuma aula iniciada ainda". Aluno com cohort mas sem `cohort_courses` populado **não** vê aulas de outros cursos (validar fallback em `src/app/(academy)/academy/(student)/curso/[slug]/page.tsx` linha 143 também aqui).
- [ ] AC2: `/academy/dashboard` "Suas turmas" lista apenas `cohort_members` com `status='ACTIVE'`. Cada card mostra `progressPercent` calculado **somente sobre lessons dos módulos liberados pela cohort** (`included_module_ids`), não sobre todas as lessons do curso. Validar via aluno com cohort restrita.
- [ ] AC3: `/academy/agenda` expõe `meeting_url` **apenas** quando `now ∈ [scheduled_at - 30min, scheduled_at + 2h]` (lógica em `src/app/(academy)/academy/(student)/agenda/page.tsx` linhas 183-185). Fora dessa janela, o campo é `null` no payload e o botão "Entrar" não renderiza. Validar via curl com cookie de sessão e inspecionar HTML.
- [ ] AC4: `ExpirationBadge` aparece quando `expires_at` está em < 30 dias; banner de aviso na dashboard (linhas 211-235) exibe link "Renovar" apontando para `/academy/perfil`. Aluno sem cohort com `expires_at` próximo não vê banner. Aluno com cohort EXPIRED não aparece em "Suas turmas".
- [ ] AC5: `/academy/meus-cursos` (página existente) lista os cursos do aluno consolidando todas as cohorts ativas. Cursos cujos módulos estão totalmente fora de `included_module_ids` não aparecem como "acessíveis" — preferencialmente são ocultados ou marcados como "trancado". Edge case: aluno com 2 cohorts no mesmo curso vê union dos módulos liberados.

## Escopo

**IN:**
- `src/app/(academy)/academy/(student)/dashboard/page.tsx`
- `src/app/(academy)/academy/(student)/meus-cursos/page.tsx`
- `src/app/(academy)/academy/(student)/agenda/page.tsx`
- `src/components/student/ExpirationBadge.tsx`
- Teste manual com 3 perfis: aluno-1-cohort-completa, aluno-1-cohort-restrita, aluno-2-cohorts-mesmo-curso

**OUT:**
- Redesign visual das páginas
- Adicionar novos widgets (notificações, leaderboard)
- Otimização de queries N+1 (anotar mas não corrigir)

## Contexto Técnico
- Dashboard faz 4 queries paralelas sobre `cohort_members`, `cohort_courses`, `lessons`, `lesson_progress`. Risco de inconsistência se uma falha silenciosa.
- `moduleIdsByCohort` é construído a partir de `cohort_courses.included_module_ids` (array). Se `null`, módulo conta como excluído (verificar comportamento esperado).
- Agenda usa `inWindow` para mascarar `meeting_url` no servidor — boa prática que precisa permanecer.
- `ProgressBar` em meus-cursos espera `value: number` (0-100).
- Performance: cada cohort = 4 queries; aluno com 5+ cohorts pode sentir lag.

### CONCERNs (não bloqueiam veredicto)

- **[CONCERN-1 perf]** `meus-cursos/page.tsx` faz 6 queries sequenciais (l.15-90). Para aluno com poucas cohorts é OK; com 5+ cohorts e dezenas de módulos, latência pode crescer. Anotado mas fora de escopo (story explicitamente lista "Otimização de queries N+1 (anotar mas não corrigir)" em OUT). **Recomendação:** considerar paralelização das fetches independentes (cohort_courses + courses + modules) numa próxima story de hardening.
