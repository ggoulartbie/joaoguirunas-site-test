# Story 2.3: Preview de markdown/HTML/MDX no editor de aula admin

## Objetivo
Permitir que o admin **veja como `description` e `content` da aula vão renderizar para o aluno** dentro do próprio editor (`/academy/admin/cursos/[courseId]/aulas/[lessonId]`), sem precisar salvar e abrir a página do aluno em outra aba. Reusar o renderer `LessonContent` já existente para garantir paridade visual perfeita.

## Acceptance Criteria

- [ ] **AC1 (Preview do `content` — já parcial)**: Manter o toggle "Preview como aluno" da seção "Conteúdo da Aula" em `LessonEditorClient.tsx:262-272`. Quando ativo, esconde o `ContentEditor` e renderiza `<LessonContent content={previewContent} className="prose prose-invert prose-sm max-w-none" />` (linha 277) com o conteúdo do estado local — **sem precisar salvar**.

- [ ] **AC2 (Preview do `description`)**: Adicionar toggle análogo "Editar / Visualizar" no campo `description` (`LessonEditorClient.tsx:186-195`). Quando em "Visualizar", renderiza o `description` como markdown via `<LessonContent content={{format: 'MARKDOWN', raw: description}} />`. Quando em "Editar", mostra o `<input>` atual. Default: "Editar".

- [ ] **AC3 (Cobertura dos 3 formatos)**: O preview de `content` cobre os 3 valores de `content_format`:
  - `MARKDOWN` → render client-side com `react-markdown` (já funciona via `LessonContent`, montar `{format: 'MARKDOWN', raw: content}`).
  - `HTML` → **sanitizar antes** de passar para `LessonContent`. Hoje em `LessonEditorClient.tsx:102` o HTML é injetado sem sanitização — risco de XSS no preview do próprio admin (paste de `<script>` por engano). Criar server action `previewContentAction(format, content)` que chama `renderContent()` de `src/lib/content/index.ts` (server-only) e devolve o `RenderedContent` tagged. Client invoca essa action ao entrar no preview.
  - `MDX` → renderizar via `renderContent('MDX', content)` server action (serialização real com `serializeMDX`). Substitui o stub quebrado em `ContentEditor.tsx:115-130` que apenas mostra o source em `<pre>`.

- [ ] **AC4 (Live preview debounced)**: Enquanto o admin digita E o preview está ativo, atualizar o `previewContent` automaticamente após **400ms** de inatividade. Usar `useEffect` + `setTimeout` cleanup. Quando o preview está fechado, **não** chama a action (evita custo desnecessário).

- [ ] **AC5 (Loading state)**: Enquanto a server action `previewContentAction` está pendente (especialmente para MDX que serializa), exibir indicador discreto "Renderizando…" no canto do painel de preview. Evitar "flash" de conteúdo vazio entre updates — manter o último preview renderizado até chegar o novo.

- [ ] **AC6 (Erro de render — MDX inválido)**: Se `serializeMDX` lança (sintaxe MDX inválida, componente desconhecido), o preview exibe **fallback amigável** com a mensagem de erro do MDX (`error.message` truncada a 200 chars) ao invés de crashar o editor. Erro NÃO impede o admin de continuar editando.

- [ ] **AC7 (Sem auto-save)**: Entrar no preview **não** chama `updateLesson`. O admin continua precisando clicar "Salvar Aula" para persistir. Preview é puro display do estado local em memória.

- [ ] **AC8 (Acessibilidade)**: Os toggles "Editar / Visualizar" são `<button type="button">` (já é o padrão do editor), com `aria-pressed` refletindo o estado atual. Foco visível com `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-[var(--ember)]`.

- [ ] **AC9 (Smoke manual)**: Como admin, em aula de teste com `content_format = 'MARKDOWN'`:
  1. Escrever `# Hello\n\n**bold** e \`code\`` no `content`.
  2. Clicar "Preview como aluno" — ver heading, negrito e inline code renderizados.
  3. Mudar para `content_format = 'HTML'`, colar `<p>Olá <strong>mundo</strong></p>` — ver renderizado.
  4. Mudar para `content_format = 'MDX'`, colar `<Callout kind="tip" title="X">conteúdo</Callout>` — ver Callout renderizado.
  5. Inserir MDX inválido (`<UnknownComponent />`) — ver fallback de erro, editor segue funcional.
  6. No `description`, digitar `Texto com **bold**` e alternar para "Visualizar" — ver `bold` renderizado em negrito.

## Escopo

- **Server action nova** (`sites-dev-alpha`): criar `previewContentAction(format: ContentFormat, content: string): Promise<RenderedContent>` em `src/app/(academy)/academy/(admin)/admin/cursos/actions.ts` (ou arquivo dedicado `preview-actions.ts`). Faz `await requireAdmin()` e chama `renderContent(format, content)` de `@/lib/content`. Retorna o `RenderedContent` tagged.
- **`LessonEditorClient.tsx`** (`sites-dev-alpha` + `sites-ux`):
  - Refatorar `handlePreview` (linhas 96-106) para chamar `previewContentAction` em vez de montar `{format, html}` cru — corrige risco XSS do HTML preview e habilita MDX real.
  - Adicionar `useEffect` debounced (400ms) que re-chama a action quando `content`/`contentFormat` mudam **enquanto** `preview === true`.
  - Adicionar toggle "Editar / Visualizar" no `description` com state local `descriptionPreview: boolean`. Em modo preview, renderizar `<LessonContent content={{format: 'MARKDOWN', raw: description}} />`. Não precisa server action — markdown puro client-side funciona.
  - Indicador de loading "Renderizando…" + error boundary leve para o painel de preview do `content` (try/catch na action, exibe fallback).
- **Cleanup do stub MDX quebrado** (`sites-dev-alpha`): remover o `MDXPreviewClient` em `src/components/editor/ContentEditor.tsx:115-130` e o uso na linha 80-94 (preview MDX agora é responsabilidade do `LessonEditorClient`, não do `ContentEditor` que é mais primitivo).

**OUT:**
- Split layout side-by-side (editor esquerda + preview direita simultâneo). Toggle "Editar / Visualizar" é mais simples em mobile e mantém padrão atual.
- Preview do campo `title` (input simples, não há markdown ali).
- Preview do video/iframe (já há embed inline em `LessonEditorClient.tsx:230-255`).
- Preview de materials (escopo de Story 2.2 / não-objetivo aqui).
- Auto-save ao trocar para preview.
- Persistir preferência "preview default ligado" entre sessões.
- Diff visual vs. estado salvo no banco ("você tem alterações não salvas").
- Reutilizar este preview na página pública do aluno (essa já é a renderização real, não é preview).
- Migration ou mudança de schema.
- Touch em `TipTapEditor` (formato HTML já é WYSIWYG — para HTML o preview é redundante mas mantemos para consistência, sanitizado).

**Dependências disponíveis (já no `package.json`):**
- `react-markdown@10.1.0`
- `next-mdx-remote@6.0.0`
- `sanitize-html@2.17.3` (usado em `src/lib/content/html.ts`)

**Decisão de arquitetura — server action vs. client puro:**
- Cliente puro com `react-markdown` direto economiza um round-trip e dá preview instantâneo, MAS:
  - Não cobre MDX (precisa `serializeMDX` server-side de `next-mdx-remote/serialize`).
  - Não sanitiza HTML (duplicaria lógica de `src/lib/content/html.ts`).
  - Diverge da renderização real do aluno (risco de "preview enganador").
- **Server action reusando `renderContent`** garante paridade 1:1 com o aluno. Custo: ~50-200ms por chamada. Debounce 400ms já mitiga.
- O `description` é exceção: campo curto, sempre markdown — pode renderizar client-side direto via `LessonContent` com `{format: 'MARKDOWN', raw: description}` sem server action.

**Risco XSS no estado atual** (`LessonEditorClient.tsx:102`):
```ts
const raw: RenderedContent =
  contentFormat === 'HTML'
    ? { format: 'HTML', html: content }   // ⚠ sem sanitização
    : { format: 'MARKDOWN', raw: content }
```
Apenas o próprio admin vê e o conteúdo só sai sanitizado quando passa por `renderContent` server-side ao ser servido ao aluno — então não é exploitable em produção. Mas paste acidental de `<script>` pode quebrar o editor. Server action resolve.

**Coordenação:**
- `sites-dev-alpha`: server action + lógica de debounce/loading/error.
- `sites-ux`: revisão visual dos toggles "Editar/Visualizar" (paridade KV com o "Preview como aluno" existente — botão `font-mono text-[10px]` minimalista, mesma família do botão linha 264-272).
