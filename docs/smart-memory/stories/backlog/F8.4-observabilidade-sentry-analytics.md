# Story F8.4: Sentry + Vercel Analytics

## Objetivo

Instrumentar a plataforma com Sentry (erros + traces) e Vercel Analytics (Web Vitals + Speed Insights).

## Acceptance Criteria

- [ ] AC1: `@sentry/nextjs` instalado e configurado para client + server + edge
- [ ] AC2: `sentry.client.config.ts` com tracing configurado (sampleRate 0.1)
- [ ] AC3: `sentry.server.config.ts` configurado; webhook handlers e Server Actions críticas marcadas com tags
- [ ] AC4: `Sentry.captureException` em todos catch blocks de Server Actions e Route Handlers
- [ ] AC5: Source maps enviados no build (Vercel integration)
- [ ] AC6: Vercel Analytics + Speed Insights ativados (já estão no site público — confirmar que continua)
- [ ] AC7: Alertas Sentry: webhook Stripe falhas (high priority); RLS errors (anom); 500 em rotas críticas
- [ ] AC8: Performance: limit de payload em context Sentry (não enviar payload completo de webhook por padrão)

## Escopo

**IN:**
- Setup Sentry
- Tags e contexts em pontos críticos
- Alertas configurados

**OUT:**
- Logging estruturado custom (Sentry breadcrumbs cobrem)
- Datadog/NewRelic (Sentry+Vercel é suficiente)

## Contexto Técnico

- ADR-001 D2.17: Sentry + Vercel Analytics
- Cuidado com PII: filtrar emails, tokens e dados sensíveis em `beforeSend`
