# Story F8.6: Deploy Vercel + produção + smoke test

## Objetivo

Subir a plataforma em produção com domínio próprio, Stripe em modo live, todas as keys de produção configuradas, e fazer smoke test pré-lançamento.

## Acceptance Criteria

- [ ] AC1: Projeto Vercel criado (ou subdomínio em projeto existente)
- [ ] AC2: Domínio configurado (ex.: `cursos.joaoguirunas.com` ou similar)
- [ ] AC3: Variáveis de ambiente de produção todas configuradas:
  - Supabase keys (mesmas do dev, projeto `mksmmpfyqowuzjcchhkl`)
  - Stripe keys de **produção** (sk_live, pk_live, webhook secret de produção)
  - Vimeo token de produção
  - Resend domínio de produção
  - `NEXT_PUBLIC_APP_URL=https://...`
  - `CRON_SECRET` para Vercel Cron auth
- [ ] AC4: Stripe webhook endpoint registrado em `https://.../api/webhooks/stripe` com eventos selecionados (checkout.session.completed, invoice.paid, invoice.payment_failed, charge.refunded)
- [ ] AC5: Vercel Cron jobs ativos (`/api/cron/daily`, `/api/cron/hourly`)
- [ ] AC6: Smoke test pré-lançamento:
  - Cadastro real → email de boas-vindas chega
  - Compra real (cartão real, valor mínimo) → liberação imediata
  - Acesso a lesson, vídeo carrega
  - Webhook processou (verificar no Stripe Dashboard + admin)
  - Refund test
  - Cron daily executa no horário (verificar próximo dia)
- [ ] AC7: README atualizado com URL de produção, links pro Stripe Dashboard, Sentry, Resend
- [ ] AC8: Backup automático Supabase ativado (Plano Pro inclui)
- [ ] AC9: Plano de rollback documentado (Vercel rollback em 1 click; Supabase migration revert SQL)

## Escopo

**IN:**
- Configuração completa de produção
- Smoke test
- Documentação de rollback

**OUT:**
- Soft launch (decisão do João, não da fase)
- Migração de dados de outras plataformas (não há)

## Contexto Técnico

- Stripe modo live: requer KYC completo da conta Stripe
- Vercel Cron: só funciona em planos Pro (verificar billing)
- Supabase backup: diário automático em Pro
- DNS: TTL baixo na primeira semana para rollback rápido
