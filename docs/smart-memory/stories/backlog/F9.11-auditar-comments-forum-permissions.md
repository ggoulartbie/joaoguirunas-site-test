# Story F9.11: Auditar comments + fórum (permissions, RLS bypass via supabaseAdmin, moderação)

## Objetivo
Garantir que comentários de aulas e threads de fórum respeitam permissões corretas: aluno só edita/deleta o próprio, MENTOR/ADMIN podem fixar e moderar, banimento bloqueia ações, e usos de `supabaseAdmin` que bypassam RLS estão justificados.

## Acceptance Criteria
- [ ] AC1: Em `/academy/curso/[slug]/aula/[lesson-slug]` (aba Comentários), aluno consegue **criar** comentário, **editar** somente os próprios e **deletar** somente os próprios (soft delete via `deleted_at`). Tentativa de editar/deletar comentário de outro aluno via `addComment`/`editComment`/`deleteComment` retorna erro de permissão (validar via DevTools forçando ID alheio). Auditar `src/lib/actions/comments.ts`.
- [ ] AC2: Usuário com `role='MENTOR'` ou `role='ADMIN'` pode **fixar** (`is_pinned=true`) e **deletar** comentários de qualquer aluno. Badge "Mentor" / "Admin" aparece corretamente nos seus comentários (já implementado em `CommentsSection.tsx`). Validar action de pin existe e respeita role check.
- [ ] AC3: Lista de comentários em `aula/[lesson-slug]/page.tsx` (linha 246-256) usa `supabaseAdmin` (bypass de RLS). Justificar: o gate `has_access` na linha 83-88 já bloqueou usuários sem acesso, então listar todos os comentários da lesson é OK. Documentar essa decisão em comentário inline ou ADR. Verificar que o mesmo padrão se aplica a `materials` (linha 275-281).
- [ ] AC4: Em `/academy/forum`, criar thread requer `cohort_member.status='ACTIVE'` (já checado linha 31-40 da forum page — redirect para `meus-cursos?erro=forum-acesso`). Validar replies têm o mesmo gate. Aluno banido (criar campo `is_banned` ou usar `status='BANNED'` em cohort_members?) não deve conseguir postar — definir mecanismo se não existir.
- [ ] AC5: Soft delete: comentários e threads marcados `deleted_at` continuam no banco mas não aparecem nas listagens (`is('deleted_at', null)` em todas queries). Comentário deletado em `CommentsSection.tsx` mostra "[comentário removido]" preservando estrutura de threads. Validar que admin tem path para hard-delete (via `/academy/admin/forum` ou `/academy/admin/moderacao`) se necessário.

## Escopo

**IN:**
- `src/lib/actions/comments.ts` — auditar `addComment`, `editComment`, `deleteComment`
- `src/components/student/CommentsSection.tsx`
- `src/app/(academy)/academy/(student)/forum/actions.ts` (e similar para `[category]/[slug]`)
- `src/app/(academy)/academy/(admin)/admin/forum/actions.ts` — moderação
- Identificação de todos os usos de `supabaseAdmin` em paths de aluno e validação caso a caso

**OUT:**
- Sistema de votos (já existe — não auditar agora)
- Notificações in-app de novos replies (escopo F8.1)
- Editor rico (Markdown / mentions @aluno)
- Reports / flags de abuso

## Contexto Técnico
- `supabaseAdmin` em path de aluno é uma decisão de design: queries de listagem/contagem onde a barreira já foi cruzada por `requireUser` + cohort gate.
- Cada action server-side deve refazer auth via `requireUser()` antes de qualquer write.
- RLS deve garantir que mesmo se uma action chamar `supabase` (não admin), só permita o user_id correto. Validar isso para `comments` e `forum_*`.
- Roles disponíveis: `STUDENT`, `MENTOR`, `ADMIN`. `MENTOR` é nivel intermediário (pode fixar comments, replicar em fórum) — checar se o RLS expressa isso.
- Cuidado com prompt injection / XSS em conteúdo de comentário — DOMPurify já é dependency conhecida (ver memory `project_vercel_deploy.md`); validar que renderização sanitiza.
