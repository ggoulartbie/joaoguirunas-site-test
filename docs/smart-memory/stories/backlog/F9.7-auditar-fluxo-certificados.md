# Story F9.7: Auditar fluxo de certificados (emissão, download PDF, verificação pública)

## Objetivo
Garantir que o fluxo de certificados está 100% funcional em produção: emissão automática ao completar curso, download de PDF a partir de dados reais (não mock), e página pública de verificação por código.

## Acceptance Criteria
- [ ] AC1: A rota `GET /api/certificado/[code]` busca o certificado **real** em `certificates` via `supabaseAdmin` (joining `profiles`, `courses`, `cohorts`) — o `TODO F5.3` no arquivo `src/app/api/certificado/[code]/route.ts` está removido e o mock substituído pela query real. Retorna 404 se o code não existir.
- [ ] AC2: Ao completar 100% das aulas de um curso (via `markLessonComplete` ou `toggleLessonComplete`), `triggerCertificateCheck` em `src/app/actions/progress.ts` invoca `checkAndIssueCertificate` que cria a linha em `certificates` com `verification_code` único, `pdf_storage_path` populado e dispara email. Verificar idempotência: completar a mesma aula duas vezes não duplica certificado.
- [ ] AC3: Página `/academy/certificados` lista todos os certificados do aluno; clicar em "Baixar PDF" abre/download do PDF real renderizado com nome do aluno, curso e data corretos. Página pública `/academy/certificado/v/[code]` exibe os dados de verificação sem exigir login.
- [ ] AC4: Edge cases cobertos por teste manual: (a) aluno sem cursos completos não vê nada na lista; (b) curso com 0 lessons não emite certificado; (c) revogação manual via `revoked_at` faz a página pública mostrar "revogado" e o PDF retornar 410 ou similar; (d) `verification_code` com caracteres especiais retorna 400 (sanitização já existe — testar).
- [ ] AC5: Rate limiting na rota pública `/api/certificado/[code]` — `renderToBuffer` é CPU-intensivo e a rota é pública sem auth, portanto vetor de DoS. Implementar limite por IP (ex.: `@upstash/ratelimit` com Vercel KV/Upstash Redis, ou Vercel Edge Config com fallback in-memory). Limite sugerido: 30 req/min por IP. Documentar a decisão (com ou sem rate limit) no runbook.
- [ ] AC6: Runbook em `docs/smart-memory/runbooks/certificados.md` documenta como reemitir manualmente um certificado, como verificar emissões em produção, variáveis de ambiente envolvidas (`NEXT_PUBLIC_APP_URL`) e configuração do rate limit (chave Upstash ou alternativa).

## Escopo

**IN:**
- `src/app/api/certificado/[code]/route.ts` — substituir mock por query real
- `src/app/actions/certificate.ts` — auditar `checkAndIssueCertificate` (idempotência, geração de code, upload de PDF para Storage)
- `src/app/actions/progress.ts` — `triggerCertificateCheck` (resolução de course_id, cohort_id correta)
- `src/app/(academy)/academy/(student)/certificados/page.tsx` — lista do aluno
- `src/app/(academy)/academy/certificado/v/[code]/page.tsx` — página pública de verificação
- Rate limiting em `/api/certificado/[code]` (P-02 da auditoria de APIs públicas — `agents/research/api-public-audit.md`)
- Runbook em `docs/smart-memory/runbooks/certificados.md`

**OUT:**
- Redesign visual do PDF (já feito em F5.3)
- Multi-idioma do PDF
- Assinatura digital / blockchain verification
- Reemissão em massa (fora do MVP)

## Contexto Técnico
- F5.3 está marcada como `done` no BACKLOG mas o código tem `TODO F5.3` — a rota retorna dados mock (`'João Guirunas'`, `'Estratégia com IA'`). Isso é um buraco crítico de produção.
- Tabela `certificates` já existe (criada em F1.4): `id, user_id, course_id, cohort_id, verification_code, issued_at, pdf_storage_path, revoked_at`.
- Geração de PDF via `@react-pdf/renderer` em `CertificatePDF` component.
- Storage bucket de certificados deve ter signed URL ou ser público read-only no path do certificado.
- Idempotência: usar `upsert` por `(user_id, course_id, cohort_id)` ou checar antes de inserir.
- Email de emissão usa Resend (via `@/lib/email/send`).
- Auditoria de APIs públicas (Rex-S, 2026-05-08) confirmou: sanitização do `code` (allowlist), PII headers e Content-Disposition estão OK. Issues abertos: P-01 (mock — coberto pelo AC1) e P-02 (rate limit — coberto pelo AC5).
