# Story 3.3: H1 duplo no DOM em /mentoria

## Objetivo
Corrigir a presença de dois `<h1>` no DOM da `/mentoria`. O layout mobile (`sm:hidden`) e desktop (`hidden sm:block`) cada um tinha seu próprio H1 — visualmente apenas um é renderizado por vez, mas crawlers veem ambos.

## Fix aplicado

O H1 mobile recebeu `aria-hidden="true"` — remove da árvore de acessibilidade e sinaliza aos crawlers que é duplicado decorativo. O H1 desktop permanece como o heading canônico.

## Acceptance Criteria

- [x] AC1: H1 no bloco `sm:hidden` tem `aria-hidden="true"`.
- [x] AC2: H1 no bloco `hidden sm:block` permanece sem `aria-hidden`.
- [x] AC3: `tsc --noEmit` passa sem erros.
- [ ] AC4: Verificar via DevTools → Elements que apenas um H1 aparece na accessibility tree.

### Issue crítico bloqueante

`aria-hidden="true"` em `<h1>` mobile (`page.tsx:132`) remove o heading principal da árvore de acessibilidade em viewport mobile. Viola WCAG 2.1:
- **1.3.1 Info and Relationships** (Level A)
- **2.4.6 Headings and Labels** (Level AA)
- **4.1.2 Name, Role, Value** (Level A)

Impacto real: iOS VoiceOver e Android TalkBack (onde SR é mais usado) não anunciarão o título da página — usuário cego perde contexto.

### Premissa da story é falsa

"H1 duplo no DOM penaliza SEO" é incorreto. Googlebot renderiza CSS desde 2019 e trata `sm:hidden` + `hidden sm:block` mutuamente exclusivos como conteúdo responsivo. Além disso, `aria-hidden` afeta **acessibilidade, não SEO** — Googlebot ignora `aria-hidden` para ranking.

### Correção exigida

Reverter `aria-hidden="true"` do `<h1>` em `page.tsx:132`. Se o objetivo for evitar H1 duplicado no HTML estático, criar story substituta com abordagem que não regrida a11y (ex.: H1 único compartilhado entre breakpoints com classes responsivas).
