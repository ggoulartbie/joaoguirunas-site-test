# Story F9.2: Auditar checkout Stripe da /curso-online (cohort, action, redirect, webhook)

## Objetivo
Garantir que a jornada de compra do produto "Curso Online" está 100% funcional em produção: cohort `curso-online-padrao` configurada com Stripe, server action redireciona para Stripe Checkout, webhook recebe `checkout.session.completed` e libera acesso (cria profile + cohort_member) ao aluno.

## Acceptance Criteria
- [ ] AC1: Existe linha em `cohorts` com `slug = 'curso-online-padrao'`, `is_purchasable = true` e `stripe_price_entry_id` populado com um Price ID válido (`price_...`) sincronizado no Stripe Dashboard. Caso não exista, a story inclui o seed/admin step para criá-la (via admin UI ou migration de seed). Documentar evidência (screenshot ou query) em QA Results.
- [ ] AC2: Clicar em "Comprar agora" em `/curso-online` (em produção e em preview) redireciona para `checkout.stripe.com/...` sem erro de servidor; quando o pagamento é concluído, retorna para `/academy/checkout/sucesso?session_id=...`.
- [ ] AC3: O webhook de Stripe (`/api/webhooks/stripe`) processa `checkout.session.completed` para esta cohort de forma idempotente, criando `payments` com `status='APPROVED'` e `cohort_members` com `status='ACTIVE'` e `expires_at` correto, e disparando email de boas-vindas.
- [ ] AC4: Se o usuário não existir, o webhook cria `auth.users` (via `supabaseAdmin.auth.admin`) e `profiles` com `role='STUDENT'`, e envia magic link / email de set-password — comprador consegue logar em `/academy/login` e ver o curso em `/academy/aluno`.
- [ ] AC5: Edge cases cobertos por teste manual e/ou e2e: (a) cohort sem `stripe_price_entry_id` retorna erro amigável; (b) cohort `is_purchasable=false` retorna erro amigável; (c) pagamento `DECLINED` não libera acesso; (d) idempotência — segundo webhook idêntico não duplica `cohort_members`.
- [ ] AC6: Runbook em `docs/smart-memory/runbooks/checkout-curso-online.md` documenta variáveis de ambiente exigidas (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_APP_URL`, `SUPABASE_SERVICE_ROLE_KEY`), passos de smoke test, e como reprocessar webhook em caso de falha.

## Escopo

**IN:**
- Auditoria do fluxo: `src/app/actions/checkoutPublic.ts`, `src/lib/payment/stripe.ts`, `src/app/api/webhooks/stripe/route.ts`
- Verificação de configuração da cohort `curso-online-padrao` em Supabase
- Teste manual e/ou Playwright cobrindo o happy path
- Runbook em `docs/smart-memory/runbooks/checkout-curso-online.md`
- Fixes pequenos descobertos durante a auditoria (código defensivo, mensagens de erro)

**OUT:**

- Adição de novos métodos de pagamento (Pix, boleto)
- Auto-renewal/cron (escopo F7.5)

## Contexto Técnico

- Server action: `src/app/actions/checkoutPublic.ts` valida cohort, `is_purchasable`, `stripe_price_entry_id`, monta sessão via `StripeAdapter` e faz `redirect(session.url)`.
- Sucesso URL: `${NEXT_PUBLIC_APP_URL}/academy/checkout/sucesso?session_id={CHECKOUT_SESSION_ID}`
- Cancel URL: `${NEXT_PUBLIC_APP_URL}/curso-online`
- O `purchaseKind: 'ENTRY'` indica que é primeira compra (não extension/renewal).
- Middleware (`src/middleware.ts`) já libera `/academy/checkout` e `/api/webhooks/*` como públicas.
- Stripe lazy-init é precondição conhecida (ver memory `project_vercel_deploy.md`).
