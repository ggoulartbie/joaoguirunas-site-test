# Story F6.4: Admin — Usuários, pagamentos, moderação, fórum

## Objetivo

Páginas admin restantes: gestão de usuários, listagem de pagamentos, moderação de comments/threads, CRUD de forum_categories.

## Acceptance Criteria

- [ ] AC1: `/admin/usuarios` tabela com filtros (cohort, status, role) + busca por nome/email
- [ ] AC2: Página de perfil do usuário no admin: memberships, progresso, comments, payments, ações (alterar role, desativar, conceder acesso a cohort, estender prazo, dar refund)
- [ ] AC3: `/admin/pagamentos` tabela com filtros (status, cohort, range de data), busca, link pro Stripe Dashboard de cada payment
- [ ] AC4: Botão "Refund" em payment APPROVED com confirmação dupla (modal + texto digitado)
- [ ] AC5: Botão "Reprocessar webhook" em payment com inconsistência (entrada manual ou triagem)
- [ ] AC6: Exportar CSV de pagamentos com filtros aplicados
- [ ] AC7: `/admin/moderacao` fila com 2 tabs: Comments novos / Threads novos (últimos 7 dias) com ações: aprovar (default), remover (soft delete), marcar spam, banir usuário
- [ ] AC8: `/admin/forum` CRUD de forum_categories (slug, name, description, sort_order, color, is_active)
- [ ] AC9: Todas as pages exigem ADMIN

## Escopo

**IN:**
- 4 páginas admin
- Refund e reprocessar webhook (handlers de baixo nível na Fase 7)
- Export CSV
- CRUD de categorias

**OUT:**
- Métricas avançadas (Fase 8)
- Auditoria de ações admin (futuro)

## Contexto Técnico

- Refund: chama Stripe API e atualiza `payments.status='REFUNDED'` + opcionalmente `cohort_members.status='REMOVED'` se for caso
- Banir: marca profile com flag `is_banned` (adicionar em ADR/migration follow-up se necessário) ou seta role para algo restrito; bloqueia login via middleware
- CSV: `Content-Type: text/csv` em Route Handler
