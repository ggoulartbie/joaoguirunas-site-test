# Story F12.1: InfinitePayAdapter + webhook handler

## Objetivo
Implementar a integração base com InfinitePay como segundo provider de pagamento da plataforma — adapter para criar links de checkout, route handler do webhook com verificação via `payment_check`, e provisionamento idempotente de `cohort_members`/`payments` ao confirmar pagamento.

## Acceptance Criteria
- [ ] AC1: `src/lib/payment/infinitepay.ts` exporta `InfinitePayAdapter` com método `createCheckoutLink({ cohortSlug, amount, customerEmail, orderId, redirectUrl, webhookUrl })` que faz `POST https://api.checkout.infinitepay.io/links` com `Authorization: Bearer ${INFINITEPAY_BEARER_TOKEN}` e `handle = INFINITEPAY_HANDLE` (env vars). Retorna `{ url }` para redirect do cliente. Lazy-init do client (mesmo pattern de `stripe.ts` — ver `project_vercel_deploy.md`).
- [ ] AC2: A server action de checkout (em F12.3) chama `createCheckoutLink` e usa Next.js `redirect(url)` para enviar o cliente ao link. Sem erro de servidor em prod e preview.
- [ ] AC3: Antes de redirecionar, persiste linha em `payments` com: `id` (UUID novo, vai como `order_nsu` no payload do InfinitePay), `cohort_id`, `user_id` (se conhecido — `null` em compra pública), `amount_cents`, `status='PENDING'`, `payment_provider='INFINITEPAY'`, `infinitepay_order_nsu` (= `id`). UUID gerado server-side via `crypto.randomUUID()`.
- [ ] AC4: `src/app/api/webhooks/infinitepay/route.ts` aceita `POST`, lê o body JSON (`invoice_slug`, `amount`, `paid_amount`, `installments`, `capture_method`, `transaction_nsu`, `order_nsu`, `receipt_url`, `items`), responde `200 OK` com `{ received: true }` em **menos de 1s** (resposta síncrona ao InfinitePay; processamento pode acontecer depois ou ser feito antes do response — desde que sub-1s). Em caso de erro interno, log via `Sentry.captureException` e responde `500` sem stack trace.
- [ ] AC5: Antes de marcar `payments` como APPROVED e provisionar acesso, valida via `POST https://api.checkout.infinitepay.io/payment_check` com `{ handle, transaction_nsu, slug: invoice_slug }` e Bearer Token. Aceita apenas se response `paid: true` e `paid_amount >= amount` (proteção contra webhook forjado, já que InfinitePay não fornece assinatura HMAC). Se `paid: false` ou erro de validação, registra evento em `webhook_events` com `processed=false` e não provisiona.
- [ ] AC6: Webhook idempotente via `order_nsu` — antes de processar, busca `payments` por `infinitepay_order_nsu = body.order_nsu`. Se já está `APPROVED`, responde `{ received: true, idempotent: true }` sem reprocessar. Se `PENDING`, processa: `payments.status → APPROVED`, cria `cohort_members` com `status='ACTIVE'`, `expires_at = now() + cohort.access_duration_days`, dispara `sendWelcomeInviteEmail`. Segundo webhook idêntico (mesma `order_nsu`) não duplica nem `cohort_members` nem nova linha em `payments`.
- [ ] AC7: `src/middleware.ts` libera `/api/webhooks/infinitepay` como rota pública (sem auth gate), igual a `/api/webhooks/stripe` e `/api/webhooks/*`. Verificar que a regex/array de rotas públicas inclui o novo path.
- [ ] AC8: `pnpm typecheck` limpo. Env vars `INFINITEPAY_HANDLE` e `INFINITEPAY_BEARER_TOKEN` adicionadas a `.env.example` com comentário explicativo. Runbook básico em `docs/smart-memory/runbooks/checkout-infinitepay.md` (env vars, fluxo do payment_check, como reprocessar webhook manualmente via `transaction_nsu`).

## Escopo

**IN:**
- `src/lib/payment/infinitepay.ts` — InfinitePayAdapter (createCheckoutLink + verifyPayment via payment_check)
- `src/app/api/webhooks/infinitepay/route.ts` — POST handler
- `src/middleware.ts` — release da rota pública
- Migration que adiciona coluna `infinitepay_order_nsu UUID` em `payments` (com índice UNIQUE para idempotência; pode ser feita junto desta story ou em F12.2 — decisão técnica, preferência por F12.1 já que o webhook depende da coluna)
- `.env.example` com novas variáveis
- Runbook em `docs/smart-memory/runbooks/checkout-infinitepay.md`

**OUT:**
- Migration `cohorts.payment_provider` (escopo F12.2)
- UI admin para escolher provider (escopo F12.2)
- Router de checkout que decide STRIPE vs INFINITEPAY (escopo F12.3)
- Suporte a parcelamento manual no admin (InfinitePay já faz nativamente via `installments` no payload)
- Reembolso/estorno via API (não-MVP)

## Contexto Técnico

- **Endpoints InfinitePay:**
  - `POST https://api.checkout.infinitepay.io/links` — cria link de checkout, retorna `{ url }`
  - `POST https://api.checkout.infinitepay.io/payment_check` — valida pagamento, retorna `{ paid: boolean, ... }`
- **Auth:** `Authorization: Bearer ${INFINITEPAY_BEARER_TOKEN}` em ambos
- **Handle:** `growthsales` (via `INFINITEPAY_HANDLE`)
- **Sem assinatura HMAC** — toda validação é via `payment_check` server-to-server. Crítico para segurança: nunca confiar no payload do webhook sem `payment_check`.
- **Payload esperado do webhook:**
  ```json
  { "invoice_slug": "...", "amount": 79700, "paid_amount": 79700, "installments": 1,
    "capture_method": "credit_card", "transaction_nsu": "...", "order_nsu": "<uuid>",
    "receipt_url": "https://...", "items": [{ "name": "...", "quantity": 1, "amount": 79700 }] }
  ```
- **Stripe pattern para reaproveitar:** `src/app/api/webhooks/stripe/route.ts` — estrutura de idempotência via `webhook_events`, criação user/profile via `findOrCreateUser`, geração magic link, `sendWelcomeInviteEmail`. Reusar essas funções, só substituir a parte de validação de assinatura por `payment_check`.
- **Tabela `payments`:** já existe com `payment_provider` (text) — confirmar em `supabase/migrations/20260506021931_*.sql`. Coluna `infinitepay_order_nsu UUID UNIQUE` precisa ser adicionada.
- **Provisionamento de user:** mesmo fluxo do Stripe — `findOrCreateUser({ email })` + magic link + welcome email.
- **`transaction_nsu` é a fonte de verdade** para verificar pagamento via `payment_check`. Nunca confiar só em `paid_amount` do webhook payload.
- **Resposta sub-1s:** considerar fazer `payment_check` + provisionamento síncrono se rápido (~300ms total esperado), ou enfileirar via `webhook_events` + processamento em background. Decisão técnica do dev.
