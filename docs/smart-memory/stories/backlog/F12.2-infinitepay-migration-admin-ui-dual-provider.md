# Story F12.2: Migration cohorts.payment_provider + admin UI dual-provider

## Objetivo
Habilitar a escolha de provider de pagamento por cohort no admin: migration adiciona coluna `payment_provider` em `cohorts` (default STRIPE, com check constraint), e o `CohortForm` ganha um dropdown para alternar entre Stripe e InfinitePay ao criar/editar uma turma.

## Acceptance Criteria
- [ ] AC1: Migration `supabase/migrations/{timestamp}_cohorts_payment_provider.sql` adiciona à tabela `cohorts`:
  - `payment_provider TEXT NOT NULL DEFAULT 'STRIPE' CHECK (payment_provider IN ('STRIPE','INFINITEPAY'))`
  - `infinitepay_handle TEXT NULL` (handle por cohort; opcional, fallback para env var `INFINITEPAY_HANDLE` se NULL)
  - Backfill: todas as cohorts existentes recebem `payment_provider='STRIPE'` (default já cobre, mas validar com SELECT pós-migration)
  - Migration **idempotente** (`ADD COLUMN IF NOT EXISTS` ou guards via `DO $$ ... $$`)
- [ ] AC2: `src/app/(academy)/academy/(admin)/admin/turmas/CohortForm.tsx` ganha um dropdown "Provider de pagamento" com opções `Stripe` e `InfinitePay`. Default `Stripe`. Salva em `cohorts.payment_provider`. Campo é editável em criação E edição (admin pode trocar provider de uma cohort existente — operação rara mas não bloqueada pela UI).
- [ ] AC3: Quando provider = `INFINITEPAY`, o form mostra um campo opcional "InfinitePay handle" que escreve em `cohorts.infinitepay_handle`. Quando vazio, comportamento esperado: backend usa env var `INFINITEPAY_HANDLE` (definido em F12.1). Quando provider = `STRIPE`, esse campo fica oculto.
- [ ] AC4: Admin consegue: (a) criar cohort nova com provider=INFINITEPAY → salva e aparece na listagem com badge correto; (b) editar cohort existente alterando provider STRIPE → INFINITEPAY → save; (c) cohort com provider=INFINITEPAY e `is_purchasable=true` aparece na lista pública de turmas/curso-online normalmente. Verificar que mudar provider numa cohort com membros ativos não corrompe membership existente (membros mantêm `cohort_member.status` independente do provider — provider só afeta novas compras).
- [ ] AC5: Migration safety protocol seguido pelo sites-data: snapshot do schema (`pg_dump --schema-only`) → dry-run em ambiente staging ou via `supabase db diff` → apply em prod com `supabase migration up` → smoke-test (SELECT contando rows pré/pós, verificar default aplicado em cohort existente, verificar check constraint rejeitando valor inválido). Documentar passos executados nos comentários da migration ou no QA Results.

## Escopo

**IN:**
- Migration nova em `supabase/migrations/`
- `src/app/(academy)/academy/(admin)/admin/turmas/CohortForm.tsx` — dropdown provider + campo handle
- Server action de save (`src/app/(academy)/academy/(admin)/admin/turmas/actions.ts`) — incluir `payment_provider` e `infinitepay_handle` no `upsert`
- Tipos TS regenerados (`pnpm supabase gen types typescript --linked > src/types/database.ts`)
- Listagem de cohorts (admin) opcionalmente exibe badge do provider — nice-to-have, não bloqueante

**OUT:**
- Lógica de routing checkout por provider (escopo F12.3)
- UI pública (LP /curso-online) refletindo provider — não muda nada visível ao aluno; provider é decidido server-side
- Stripe webhook não muda
- Migration não remove `cohorts.stripe_price_entry_id` (continua coluna válida; só usada quando provider=STRIPE)

## Contexto Técnico

- **Tabela `cohorts`** — schema atual em `supabase/migrations/20260506021931_*.sql` (criação via F1.3) e migrations subsequentes. Já tem `stripe_price_entry_id`, `is_purchasable`, `entry_price_cents`, `access_duration_days`, `slug`, etc.
- **Pattern de migration anterior:** `supabase/migrations/20260508000000_seed_curso_online_padrao.sql` é boa referência de estilo (idempotente, com comentários).
- **CohortForm:** atualmente é form React com `useState` por campo, server action `upsertCohort` chamada no submit. Manter o pattern. Dropdown via `<select>` nativo ou shadcn `Select` (se já tiver shadcn no projeto, conferir antes).
- **Decisão de produto:** provider por cohort (não global) — permite migrar gradualmente. Default Stripe garante que cohorts antigas continuam funcionando sem mudança.
- **Coluna `infinitepay_handle` opcional:** dá flexibilidade se um cliente futuro quiser handle diferente (B2B). Por ora, todas usam env var.
- **Coordenação:** sites-data faz a migration e regenera types; sites-dev-alpha faz a UI. sites-data conclui primeiro (UI depende dos types novos).
