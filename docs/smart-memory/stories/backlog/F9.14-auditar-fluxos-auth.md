# Story F9.14: Auditar fluxos auth (login, cadastro, recuperar/redefinir senha)

## Objetivo
Validar segurança e correção dos 4 fluxos de auth: `/academy/login`, `/academy/cadastro`, `/academy/recuperar-senha`, `/academy/redefinir-senha`. Foco em open redirect, validação de input, password policy, race condition em redefinição, e estado pós-auth (cookie + role).

## Acceptance Criteria
- [ ] AC1: Open redirect — query `?next=` no login só aceita paths internos (começa com `/academy/`). Tentativa de `?next=https://evil.com` redireciona para `/academy/dashboard` (default) ou rejeita.
- [ ] AC2: Cadastro — `/academy/cadastro` valida:
  - Email formato válido (Zod / HTML5)
  - Senha mínima de 8 chars (mínimo recomendado pela política do Supabase Auth — confirmar configuração)
  - Erros do Supabase (`User already registered`, etc) exibidos sem vazar stack
  - Rate limit verificado (Supabase Auth tem default — confirmar config)
- [ ] AC3: Recuperar senha — token de reset:
  - Link enviado por email expira (< 24h, conforme config Supabase)
  - Tentar usar token expirado → mensagem clara de erro
  - Tentar reusar token já consumido → bloqueia
  - Página `/academy/redefinir-senha` valida token antes de aceitar nova senha
- [ ] AC4: Pós-login — cookie de sessão tem flags `Secure`, `HttpOnly`, `SameSite=Lax`. Verificar via DevTools.
- [ ] AC5: Login com Google OAuth — fluxo completo: redirect URL whitelisted no Supabase, callback respeita `?next=` validado.
- [ ] AC6: Cadastro NÃO cria conta sem confirmação de email (config Supabase) — testar fluxo: cadastrar → tentar logar antes de confirmar → bloqueado com mensagem clara.
- [ ] AC7: Documentação em `docs/smart-memory/agents/research/auth-audit.md` com hardening recomendado (config Supabase, headers, rate limits).
- [ ] AC8: Bugs P0/P1 CORRIGIDOS na story; P2+ documentados.
